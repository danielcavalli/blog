POR QUE ANIMAÇÕES EM CSS?

23 de Outubro de 2025 • 7 min de leitura
csswebanimacoesdesempenho

Criando animações suaves e eficientes usando apenas CSS. Um mergulho profundo na construção de sistemas de movimento coerentes para a web moderna.

Por que animações em CSS?#Link to this section

Animações em CSS trazem várias vantagens em relação às animações baseadas em JavaScript. Elas são aceleradas por hardware, têm melhor desempenho e mantêm 60fps com suavidade até em dispositivos mais modestos. Mais importante, porém, é que permitem uma abordagem declarativa para o movimento: você define o que deve acontecer, e o navegador cuida de como.

Este blog foi construído inteiramente com uma abordagem CSS-first para animações. JavaScript entra apenas para disparar mudanças de estado; o comportamento visual vive nas folhas de estilo.

A filosofia do movimento#Link to this section

Antes de escolher técnicas de animação, vale pensar no que esse movimento significa. Toda transição deveria responder a três perguntas: De onde? Para onde? Como isso se relaciona com o que veio antes?

Neste site, as animações servem a três princípios centrais:
1. Continuidade: tudo existe sobre uma única superfície ininterrupta
2. Plausibilidade física: o movimento ecoa dinâmicas do mundo real sem tentar imitá-las
3. Energia contida: movimento intencional, sem agitação

Movimento nunca é decoração. Ele estabelece relações espaciais e reconhece mudanças de estado. A interface respira.

Os blocos fundamentais#Link to this section

O CSS oferece três formas principais de criar animações:

Transições#Link to this section

Interpolações suaves entre estados. São usadas em efeitos de hover, mudanças de tema e expansões de dropdown. O segredo está em usar constantes de movimento compartilhadas. Todas as transições deste site usam a mesma curva de easing: cubic-bezier(0.4, 0, 0.2, 1).

:root {
    --motion-duration-core: 500ms;
    --motion-duration-quick: 300ms;
    --motion-duration-ripple: 600ms;
    --motion-easing: cubic-bezier(0.4, 0, 0.2, 1);
}

.custom-select {
    transition: all var(--motion-duration-core) var(--motion-easing);
}

Isso cria um ritmo temporal compartilhado. Cada interação parece fazer parte da mesma coreografia.

Animações com keyframes#Link to this section

Coreografia em múltiplas etapas para movimentos mais complexos. Este blog usa keyframes com parcimônia: só nas animações de carregamento inicial da página, quando os elementos entram em fade e sobem um pouco.

@keyframes fadeInUp {
    from {
        opacity: 0;
        transform: translateY(20px);
    }
    to {
        opacity: 1;
        transform: translateY(0);
    }
}

Propriedades transform#Link to this section

A espinha dorsal do desempenho. translateY, scale e rotate são acelerados por hardware e não disparam reflows custosos. Quando o cabeçalho desliza por trás da barra de navegação, ele usa translateY(-120%) com duração de 500ms, alinhada ao tempo central do sistema de movimento.

View Transitions API#Link to this section

Navegadores modernos oferecem a View Transitions API, que permite um morphing contínuo entre navegações de página. Este blog a usa para transformar cards de posts em artigos completos:

::view-transition-old(post-container-1),
::view-transition-new(post-container-1) {
    animation-duration: var(--motion-duration-core);
    animation-timing-function: var(--motion-easing);
}

A sacada crítica é esta: carregue o conteúdo ANTES de chamar startViewTransition(). Troque o DOM de forma síncrona dentro do callback. Isso garante que o navegador capture corretamente o estado antigo e o novo, evitando piscadas de conteúdo.

async function navigateTo(url) {
    const response = await fetch(url);
    const html = await response.text();

    document.startViewTransition(() => {
        // Synchronous DOM swap here
        updateDOM(html);
    });
}

Desempenho importa#Link to this section

Nem todas as propriedades de CSS se equivalem quando o assunto é desempenho em animações.

Aceleradas por hardware (GPU):
- transform (translate, scale, rotate)
- opacity
- filter (com cautela: pode sair caro)

Evite animar:
- width, height, margin, padding (disparam layout)
- color, background-color isoladamente (disparam paint)
- box-shadow (é custoso, mas aceitável em estados sutis de hover)

Use will-change com parcimônia e apenas durante animações ativas:

.post-card.filtering-out {
    opacity: 0;
    will-change: opacity;
}

.post-card:not(.filtering-out) {
    will-change: auto; /* Reset after animation */
}

Isso sinaliza ao navegador para otimizar a composição na GPU, mas só quando necessário. Deixar will-change ativo o tempo todo desperdiça memória.

O ritmo da interação#Link to this section

Todas as animações deste site seguem um ritmo temporal consistente, definido por propriedades personalizadas do CSS:
- 500ms (core) para a maioria das transições (morphing, deslizamento, expansão)
- 300ms (quick) para microinterações (feedback de hover)
- 600ms (ripple) para mudanças atmosféricas (transições de tema)
- A mesma curva de easing em toda parte: cubic-bezier(0.4, 0, 0.2, 1)

Essa consistência dá à interface uma sensação contínua de respiração. Quando os filtros se expandem, os dropdowns se abrem e os cards se dissolvem, tudo se move no mesmo pulso.

Revelações escalonadas#Link to this section

Ao exibir vários elementos, escalone sua entrada com atrasos de transição em CSS:

.filter-row {
    transition: all var(--motion-duration-core) var(--motion-easing) 0.1s;
}

.filter-tags {
    transition: all var(--motion-duration-core) var(--motion-easing) 0.2s;
}

Os controles de filtro aparecem 100ms antes das pílulas de tag, criando uma cascata que conduz o olhar naturalmente para baixo. Essa expansão cadenciada faz a interface parecer se desdobrar a partir de uma única superfície.

Limites orgânicos#Link to this section

Um padrão particular deste blog é a acomodação por overflow. Quando uma lista de dropdown ultrapassa o limite do painel de filtros, o bloco inteiro se expande a partir da borda inferior, abrindo espaço de forma orgânica:

.filters.expanded.dropdown-active {
    padding-bottom: 5rem;
}

O JavaScript mede o overflow real do DOM com getBoundingClientRect() e aplica a classe quando necessário. O CSS cuida da expansão suave. O resultado é uma página que parece perceber a própria geometria e reagir de forma física.

Transições de tema#Link to this section

Alternar entre modo claro e modo escuro não é só trocar cores. É uma mudança atmosférica de luz. A chave está em trocar o tema PRIMEIRO e só então habilitar as transições:

function applyTheme(theme, animated) {
    if (animated) {
        // Change theme immediately
        document.documentElement.setAttribute('data-theme', theme);

        // Force reflow
        document.body.offsetHeight;

        // Enable smooth transitions for 600ms
        document.body.classList.add('theme-transitioning');

        setTimeout(() => {
            document.body.classList.remove('theme-transitioning');
        }, 600);
    }
}

body.theme-transitioning * {
    transition: background-color var(--motion-duration-ripple) var(--motion-easing),
                color var(--motion-duration-ripple) var(--motion-easing),
                border-color var(--motion-duration-ripple) var(--motion-easing);
}

O efeito é este: a página inteira ondula para um novo estado de luz em 600ms, mantendo a ilusão de uma única superfície contínua. Sem flash, sem trancos bruscos.

FLIP: animando mudanças de layout#Link to this section

Ao filtrar os posts do blog, os cards precisam se reorganizar sem trancos. A técnica FLIP (First, Last, Invert, Play) faz mudanças de layout parecerem animadas:

Coreografia em duas fases:
1. Dissolução: os cards que perdem relevância desaparecem em 500ms
2. Reorganização: os cards restantes deslizam para suas novas posições usando FLIP

// Phase 1: Capture initial positions
const initialPositions = new Map();
cards.forEach(card => {
    const rect = card.getBoundingClientRect();
    initialPositions.set(card, { top: rect.top, left: rect.left });
});

// Apply filter changes (cards removed from layout)
cardsToHide.forEach(card => card.classList.add('filtering-out'));

// Wait for dissolve to complete via transitionend event
grid.addEventListener('transitionend', () => {
    // Phase 2: Capture final positions
    const finalPositions = new Map();
    visibleCards.forEach(card => {
        const rect = card.getBoundingClientRect();
        finalPositions.set(card, { top: rect.top, left: rect.left });
    });

    // FLIP: Calculate deltas and animate
    visibleCards.forEach(card => {
        const deltaY = initial.top - final.top;
        const deltaX = initial.left - final.left;

        card.style.setProperty('--flip-x', `${deltaX}px`);
        card.style.setProperty('--flip-y', `${deltaY}px`);
        card.classList.add('flip-animating');
    });
});

.post-card.flip-animating {
    transition: transform var(--motion-duration-core) var(--motion-easing);
    transform: translate(var(--flip-x, 0), var(--flip-y, 0));
    will-change: transform;
}

O resultado é este: os cards se dissolvem, depois o grid se contrai com suavidade. Sem saltos bruscos. JavaScript controla o estado; CSS controla o visual.

Exemplos práticos#Link to this section

Toda interação neste blog demonstra esses princípios:

  • Card hover: elevação com translateY(-2px) e abertura da sombra (300ms quick)
  • Pílulas de tag: fundos em gradiente com elevação de 3px e sombras em expansão (500ms core)
  • Abertura do dropdown: max-height: 0 a 300px com fade de opacidade (500ms)
  • Expansão do filtro: revelações escalonadas (atrasos de 100ms e 200ms) com crescimento a partir da borda inferior
  • Morphing do post: View Transitions API para a transformação de card em artigo (500ms)
  • Ripple do tema: todas as propriedades de cor transitam juntas (mudança atmosférica de 600ms)
  • Filtragem de cards: dissolução em duas fases (500ms) e depois reorganização com FLIP (500ms)

Cada animação responde: De onde? Para onde? Como isso se relaciona com o que veio antes? Nada aparece ou desaparece sem contexto espacial.

Acessibilidade em primeiro lugar#Link to this section

Sempre respeite as preferências do usuário com a media query prefers-reduced-motion. Para algumas pessoas, animações distraem ou desorientam:

@media (prefers-reduced-motion: reduce) {
    * {
        animation-duration: 0.01ms !important;
        transition-duration: 0.01ms !important;
    }
}

O movimento deve ajudar, nunca atrapalhar. Quando o usuário sinaliza que precisa de movimento reduzido, isso deve ser respeitado imediatamente.

Conclusão#Link to this section

Animações em CSS, quando tratadas de forma sistemática, produzem interfaces que parecem vivas, mas calmas. O objetivo não é impressionar com transições espalhafatosas, e sim construir confiança por meio da coerência espacial. O ponto nunca foi só desempenho, embora a interface também deva passar sensação de desempenho justamente pela ideia por trás dessas transições: fluidez. Uma transição truncada enfraquece a interface. Seria possível implementar tudo isso em JavaScript puro, mas eu não sou um bom programador de JS, nem quero ser. Apoiar-se pesadamente nos recursos de CSS para abstrair a lógica de implementação acelera o desenvolvimento e ainda transfere para o próprio navegador parte da responsabilidade pela otimização de desempenho.

Toda animação neste blog serve à mesma filosofia:
- Continuidade: uma única superfície ininterrupta
- Plausibilidade física: movimento que ecoa dinâmicas reais
- Energia contida: intencional, nunca agitada

Use propriedades personalizadas do CSS para compartilhar timings. Deixe o navegador cuidar do sequenciamento via eventos transitionend. Aplique will-change apenas durante animações ativas. Sempre pergunte: De onde? Para onde? Como isso se relaciona com o que veio antes?

A interface não se exibe. Ela responde.
