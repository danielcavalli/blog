O SISTEMA ESTÁTICO VIVO

25 de Janeiro de 2025 • 11 min de leitura
Desenvolvimento WebView Transitions APISites EstáticosPython

Por dentro de um site estático bilíngue que se move como uma única superfície: um build em Python, um pipeline de tradução e transições nativas que mantêm o espaço contínuo.

O sistema começou como uma pergunta: o quão pouco um blog precisa para parecer vivo. Não animado por si só, mas vivo da mesma forma que a maré avança: inevitável, contínuo, sem bordas. O que surgiu foi um gerador estático que escreve dois espelhos de si mesmo, um em inglês e um em português, e uma camada fina de movimento que nunca quebra a superfície. As páginas não dão um switch. O espaço se reorganiza.

O Pipeline de Build

O build é uma única etapa de trabalho deliberado. O Markdown entra por uma porta estreita. O front matter é lido, o conteúdo é processado e um modelo de página é montado com estrutura suficiente para renderizar precisamente o que é necessário e nada mais. O script não orquestra um framework. Ele compõe arquivos.

Dentro de _source/build.py, os posts são descobertos ao percorrer o diretório _source/posts. Cada arquivo markdown é processado com a biblioteca frontmatter, que separa o metadata em YAML do conteúdo do corpo. O corpo flui através da biblioteca markdown do Python com extensões para tabelas e blocos de código. O que sai do outro lado é um HTML limpo envolvido em um template que sabe exatamente onde o título vai, onde a data fica e como a navegação deve respirar.

O único cache que importa é aquele que respeita a mudança. Um hash de conteúdo é computado a partir da fonte em inglês usando MD5, e qualquer transformação que venha a seguir herda essa decisão. O hash vive no próprio front matter, gravado de volta no arquivo markdown após o processamento:

def parse_markdown_post(filepath):
    post = frontmatter.load(filepath)
    content_hash = calculate_content_hash(post.content)

    stored_hash = post.get('content_hash')
    needs_update = False

    if not created_at:
        # Post novo - define a data de criação
        post['created_at'] = now
        post['content_hash'] = content_hash
        needs_update = True
    elif stored_hash != content_hash:
        # Conteúdo alterado - atualiza o timestamp
        post['updated_at'] = now
        post['content_hash'] = content_hash
        needs_update = True

Quando nada mudou, nada sofre rebuild. Quando uma frase muda, o pipeline acorda e se move com ela. O script de build escreve em dois diretórios de saída, /ene /pt, ambos na raiz do projeto. Ambos são sites completos. Ambos são de primeira classe. Não existe fallback nem busca em runtime.

A Tradução como um Espelho

A tradução não é uma decoração. É uma segunda superfície desenhada pela mesma mão. O inglês entra; o português sai através do Gemini com restrições que preservam blocos de código, terminologia técnica e o tom do original.

O tradutor é inicializado com uma chave de API do ambiente. Ele mantém um arquivo de cache JSON em _cache/translation-cache.json, indexado pelo slug de cada post e pelo hash do conteúdo. A regra é simples: se a fonte não mudou, a tradução já é válida. Se o hash for diferente, uma nova tradução é solicitada e o cache é atualizado de forma atômica.

O prompt de tradução não é uma instrução simples. É um contrato:

Você é um redator técnico brasileiro bilíngue traduzindo um post de blog do inglês para um português brasileiro natural e localizado. Sua tarefa não é fazer uma tradução literal, mas localizar o texto — transformando-o em algo que pareça ter sido originalmente escrito em português pelo mesmo autor.

O prompt continua com princípios: preservar termos técnicos que os brasileiros usam em inglês (framework, build, pipeline, deploy), mas localizar o conteúdo narrativo e pessoal para soar fluente e culturalmente preciso. O tradutor respeita a essência do original enquanto faz cada frase respirar naturalmente no segundo idioma.

translated = translator.translate_post(
    slug, frontmatter, content
)

O resultado é armazenado no cache, e o próximo build pula a chamada de API inteiramente. É assim que um site bilíngue pode fazer o rebuild em segundos em vez de minutos. O cache é a memória do que já foi dito.

Navegação como um Espaço Contínuo

A navegação se comporta como se o documento fosse um plano único que aprende novas formas. No navegador, a View Transitions API não é um efeito especial. É uma memória do antes e do depois. O novo HTML é buscado antecipadamente. O swap acontece dentro de um único callback síncrono e o navegador captura dois estados do mesmo espaço. O código é pequeno porque o navegador está fazendo o que foi projetado para fazer.

A função navigateTo em static/js/transitions.js intercepta cliques em links internos. Ela busca a nova página, a transforma em um fragmento de documento e prepara o swap. Antes da transição começar, ela remove os atributos view-transition-name dos cards de posts no novo documento. Isso evita que o navegador crie pseudo-elementos extras que poderiam piscar durante a limpeza. Então a transição começa:

const transition = document.startViewTransition(() => {
  document.title = newDoc.title;

  const main = document.querySelector('main');
  const newMain = newDoc.querySelector('main');
  if (main && newMain) main.replaceWith(newMain);

  history.pushState(null, '', url);
  window.scrollTo({ top: 0, behavior: 'instant' });
});

await transition.finished;

A coreografia é escrita em CSS, não em JavaScript. O código acima apenas decide quando a superfície deve se lembrar. Todo o resto vive nos estilos: curvas de easing compartilhadas entre interações, durações que rimam e pseudo-elementos ::view-transition que permitem que os títulos se transformem sem nunca desaparecer. As constantes de movimento ficam em um só lugar e não negociam com o resto do design:

:root {
  --motion-duration-core: 500ms;
  --motion-duration-quick: 300ms;
  --motion-duration-ripple: 600ms;
  --motion-easing: cubic-bezier(0.4, 0, 0.2, 1);
}

Cada interação no site usa essas variáveis. Os cards de posts levantam no hover por 300ms. As transições de tema se propagam por 600ms. As transições de visualização se transformam por 500ms. A consistência não é um acidente. É o som do mesmo relógio batendo em todos os lugares.

Filtragem Sem Surpresas

Os cards na página inicial aprendem a se comportar como bons cidadãos. O primeiro dever deles é aparecer, depois ficar parados. A ordenação e a filtragem são feitas com FLIP, não com surpresas. FLIP significa First, Last, Invert, Play: meça onde cada card começa, aplique a regra de filtragem ou ordenação, meça onde cada card termina e depois anime o delta.

A implementação em static/js/filter.js captura os retângulos delimitadores iniciais de todos os cards visíveis, aplica a lógica de filtragem para esconder ou mostrar os cards e depois captura as posições finais. Cada card que se moveu recebe uma transformação CSS para invertê-lo de volta à sua posição inicial, e então a transformação é removida ao longo da duração da animação. O card parece deslizar suavemente da posição antiga para a nova.

const rects = new Map();
cards.forEach(card => {
  rects.set(card, card.getBoundingClientRect());
});

// Aplica o filtro (esconde/mostra cards, o layout se reorganiza)
applyFilter();

cards.forEach(card => {
  const oldRect = rects.get(card);
  const newRect = card.getBoundingClientRect();
  const dx = oldRect.left - newRect.left;
  const dy = oldRect.top - newRect.top;

  card.style.transform = `translate(${dx}px, ${dy}px)`;
  requestAnimationFrame(() => {
    card.style.transition = 'transform 500ms var(--motion-easing)';
    card.style.transform = 'none';
  });
});

O grid nunca dá saltos. Ele exala. Quando a navegação chega de outro lugar, o keyframe inicial de revelação é totalmente ignorado. O script detecta a navegação verificando o document.referrer para a mesma origem. Se o referrer bater, ele assume que o usuário navegou de outra página do site e desativa imediatamente a animação de introdução adicionando disable-animation ao grid de posts e definindo animation: none em cada card. O resultado é silencioso: nada pisca, nada reseta, nada se impõe após o fato.

Troca de Idioma como uma Camada de Tradução

Uma das continuidades mais sutis do sistema é como a troca de idioma é sentida. Quando você clica no seletor de idioma, você não está saindo da página. Você está traduzindo a superfície que já está lendo. A posição do scroll é preservada. O estado do filtro é preservado. A URL muda de /en/index.htmlpara/pt/index.html, mas o seu lugar no documento não muda.

Isso é gerenciado no transitions.js detectando uma troca de idioma: a estrutura do caminho é idêntica, exceto pelo segmento en ou pt. Se esse padrão for detectado, a posição do scroll é capturada antes da busca e restaurada após o swap do DOM. Se você estava no meio de um artigo longo em inglês e muda para o português, você continua no meio do mesmo artigo. O conteúdo muda, mas o espaço não.

O mesmo princípio se estende ao estado do filtro na página inicial. Se você filtrou os posts por ano ou tag e troca de idioma, o filtro permanece ativo. Os posts traduzidos aparecem, mas os critérios de seleção são preservados. É assim que um site bilíngue parece um único documento visto através de duas lentes, em vez de dois sites separados.

Customização Sem Estruturas Complexas

A customização respeita a essência do sistema. O ponto de entrada é a configuração e o texto, não o scaffolding. Abra _source/config.py e você encontrará tudo o que pode mudar: caminho base, definições de idioma, metadata do site e strings de interface para ambos os idiomas. Mude o caminho base e o build escreve links que obedecem. Ajuste variáveis tipográficas em static/css/styles.css e toda a superfície se ajusta uniformemente.

O conjunto de idiomas é a única suposição fixa. O inglês é a fonte da verdade. O português é uma tradução fiel. Se um terceiro idioma for necessário, a regra se estende: adicione um segundo espelho, dê a ele um cache, mantenha os hashes honestos. O construtor é um script que você pode ler em uma única sentada. Se você consegue lê-lo, você consegue mudá-lo. Não há mágica. Não há camadas ocultas. O sistema é exatamente tão complexo quanto o problema que ele resolve.

Por que Nenhum Framework

Recusar um framework aqui não é nostalgia. É um alinhamento com o problema. Um blog é texto, algumas imagens e a sensação de que se mover por ele deve ser como virar uma página sem perder a linha que você estava lendo. As capacidades nativas do navegador já são suficientes. Python já é suficiente. As dependências são escolhidas como ferramentas em uma bancada pequena: se uma ferramenta não faz o corte ser mais limpo, ela fica na gaveta.

A ausência de maquinário não é uma ideologia. É manutenção praticada com antecedência. Cada linha de código neste sistema é um código que terei que ler novamente em seis meses ou seis anos. Cada dependência é uma potencial incompatibilidade futura. Cada abstração é um imposto sobre a compreensão. Então o sistema permanece pequeno. Permanece legível. Permanece mutável.

A View Transitions API é nativa do Chromium. Não requer polyfill, nem wrapper, nem etapa de build. O navegador já sabe como capturar dois estados do DOM e animar entre eles. Meu trabalho é decidir quando isso deve acontecer e evitar que o resto da página interfira. O CSS que descreve o movimento é declarativo. Ele parece uma partitura de coreografia, não um programa.

As bibliotecas frontmatter e markdown do Python fazem exatamente uma coisa cada: processar YAML e converter Markdown para HTML. Elas fazem isso bem. Elas não tentam ser sistemas de build. Elas não tentam ser roteadores. Elas processam texto. Isso é tudo o que eu preciso que elas façam.

A Filosofia do Movimento

A filosofia do movimento é a filosofia da atenção. Nada deve ser insistente. As transições respondem a três perguntas: de onde, para onde e como este estado se relaciona com o anterior. Mudanças de tema se propagam como uma nuvem passando pelo sol. Um card sobe alguns pixels quando você passa o mouse e depois volta ao repouso. A superfície nunca mente sobre o que se moveu ou por quê.

Quando algo deve ser removido, ele se dissolve ao longo de 500 milissegundos. Quando algo deve chegar, emerge de onde poderia plausivelmente estar. O card do post na página inicial se expande para o artigo completo. O título permanece no lugar e cresce. A data se ancora no canto. O layout respira para fora sem quebrar a continuidade. Quando você pressiona "voltar", o artigo se contrai no card novamente. O espaço é conservado.

A curva de easing é a mesma em todos os lugares: cubic-bezier(0.4, 0, 0.2, 1). Esta não é uma escolha aleatória. É o easing padrão do Material Design, escolhido porque parece natural para a percepção humana. O movimento começa com intenção, acelera suavemente e desacelera ao completar. Nada dá solavancos. Nada desliza em velocidade constante. Tudo se move como se tivesse peso.

O Contrato com o Leitor

Para aqueles que querem adaptá-lo, o contrato é pequeno e explícito. Markdown entra, HTML sai. Um tradutor que respeita o código e a voz. Uma camada de navegação que nunca gera um bloqueio na renderização com trabalho que poderia ter sido feito antes. Estilos que codificam o ritmo uma vez e o reusam em todos os lugares. Se você mudar uma coisa, deve parecer que mudou exatamente uma coisa e o resto do sistema deve responder com uma calma proporcional.

A única esperteza está em saber quando parar. O sistema poderia fazer mais. Poderia ter um CMS. Poderia ter preview em tempo real. Poderia ter analytics, testes A/B e widgets de compartilhamento social. Ele não faz nada disso porque nada disso melhora a experiência de leitura. O objetivo não é construir uma plataforma. O objetivo é escrever e deixar a escrita ser lida sem interferência.

O Que o Sistema Sabe

Isso não é uma demonstração. É a versão mais simples de um documento vivo que eu poderia fazer sem mentir sobre o que a web já consegue fazer. Na maioria das noites, escrevo com as janelas abertas em Copacabana e o som da avenida se misturando ao mar. O sistema aprende esse ritmo e o mantém. As coisas fluem não porque são rápidas, mas porque não ficam no próprio caminho.

O cache lembra o que não mudou. O tradutor preserva o que não deve ser traduzido. A navegação mantém para onde você estava olhando quando o idioma mudou. O sistema de movimento codifica quanto tempo as coisas devem levar e se recusa a discutir sobre isso. Cada decisão que o sistema toma é em serviço de uma única ideia: o documento é contínuo. Quando você se move através dele, o espaço se reorganiza, mas nunca se quebra.

É isso que significa para um site estático parecer vivo.
