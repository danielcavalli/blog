O SISTEMA ESTÁTICO VIVO

25 de Janeiro de 2025 • 11 min de leitura

O sistema começou como uma pergunta: quão pouco um blog precisa para parecer vivo. Não animado por si só, mas vivo da forma como a maré avança: inevitável, contínuo, sem bordas.

O sistema começou como uma pergunta: quão pouco um blog precisa para parecer vivo. Não animado por si só, mas vivo da forma como a maré avança: inevitável, contínuo, sem bordas. O que surgiu é um gerador estático que escreve dois espelhos de si mesmo, um em inglês e outro em português, e uma fina camada de movimento que nunca rompe a superfície. As páginas não trocam. O espaço se reorganiza.

O Pipeline de Build

O build é uma única passagem de trabalho deliberado. O Markdown entra por uma porta estreita. O front matter é lido, o conteúdo é analisado e um modelo de página é montado com estrutura suficiente para renderizar exatamente o que é necessário, e nada mais. O script não orquestra um framework. Ele compõe arquivos.

Dentro de _source/build.py, os posts são descobertos percorrendo o diretório _source/posts. Cada arquivo Markdown é analisado com a biblioteca frontmatter, que separa os metadados YAML do corpo do conteúdo. O corpo passa pela biblioteca markdown do Python com extensões para blocos de código delimitados e tabelas. O que sai do outro lado é HTML limpo, envolvido em um template que sabe exatamente onde o título vai, onde a data fica e como a navegação deve respirar.

O único cache que importa é o que respeita mudança. Um hash de conteúdo é calculado a partir da fonte em inglês usando MD5, e qualquer transformação que venha depois herda essa decisão. O hash vive no próprio front matter, escrito de volta no arquivo Markdown depois da análise:

def parse_markdown_post(filepath):
    post = frontmatter.load(filepath)
    content_hash = calculate_content_hash(post.content)

    stored_hash = post.get('content_hash')
    needs_update = False

    if not created_at:
        # New post - set creation date
        post['created_at'] = now
        post['content_hash'] = content_hash
        needs_update = True
    elif stored_hash != content_hash:
        # Content changed - update timestamp
        post['updated_at'] = now
        post['content_hash'] = content_hash
        needs_update = True

Quando nada mudou, nada é reconstruído. Quando uma frase se move, o pipeline desperta e se move com ela. O script de build escreve em dois diretórios de saída, /en e /pt, ambos na raiz do projeto. Ambos são sites completos. Ambos são de primeira classe. Não há fallback nem fetch em runtime.

Tradução Como Espelho

Tradução não é decoração. É uma segunda superfície desenhada pela mesma mão. O inglês entra; o português sai pelo Gemini com restrições que preservam blocos de código, terminologia técnica e o tom do original.

O tradutor é inicializado com uma chave de API do ambiente. Ele mantém um arquivo de cache JSON em _cache/translation-cache.json, indexado pelo slug de cada post e pelo hash do conteúdo. A regra é simples: se a fonte não mudou, a tradução já é verdadeira. Se o hash for diferente, uma nova tradução é solicitada e o cache é atualizado de forma atômica.

O prompt de tradução não é uma instrução simples. É um contrato:

You are a bilingual Brazilian technical writer translating an English blog post into natural, localized Brazilian Portuguese. Your task is not to perform a literal translation but to localize the text—transforming it into something that reads as if it were originally written in Portuguese by the same author.

O prompt continua com princípios: preservar termos técnicos que brasileiros usam em inglês (framework, build, pipeline, deploy), mas localizar conteúdo narrativo e pessoal para soar fluente e culturalmente preciso. O tradutor respeita o grão do original enquanto faz cada frase respirar naturalmente na segunda língua.

translated = translator.translate_post(
    slug, frontmatter, content
)

O resultado vai para o cache, e o próximo build pula completamente a chamada de API. É assim que um site bilíngue pode ser reconstruído em segundos em vez de minutos. O cache é a memória do que já foi dito.

Navegação Como Espaço Contínuo

A navegação se comporta como se o documento fosse um único plano que aprende novas formas. No navegador, a View Transitions API não é um efeito especial. É uma memória do antes e do depois. O novo HTML é buscado antecipadamente. A troca acontece dentro de um único callback síncrono e o navegador captura dois estados do mesmo espaço. O código é pequeno porque o navegador está fazendo o que foi projetado para fazer.

A função navigateTo em static/js/transitions.js intercepta cliques em links internos. Ela busca a nova página, a analisa em um fragmento de documento e prepara a troca. Antes de a transição começar, ela remove atributos view-transition-name dos cards de post no novo documento. Isso impede o navegador de criar pseudo-elementos extras que piscariam durante a limpeza. Depois a transição começa:

const transition = document.startViewTransition(() => {
  document.title = newDoc.title;

  const main = document.querySelector('main');
  const newMain = newDoc.querySelector('main');
  if (main && newMain) main.replaceWith(newMain);

  history.pushState(null, '', url);
  window.scrollTo({ top: 0, behavior: 'instant' });
});

await transition.finished;

A coreografia é escrita em CSS, não em JavaScript. O código acima só decide quando a superfície deve se lembrar. Todo o resto vive nos estilos: curvas de easing compartilhadas entre interações, durações que rimam e pseudo-elementos ::view-transition que deixam títulos se transformarem sem jamais desaparecer. As constantes de movimento ficam em um só lugar e não negociam com o resto do design:

:root {
  --motion-duration-core: 500ms;
  --motion-duration-quick: 300ms;
  --motion-duration-ripple: 600ms;
  --motion-easing: cubic-bezier(0.4, 0, 0.2, 1);
}

Toda interação no site usa essas variáveis. Cards de post se elevam no hover por 300ms. Transições de tema ondulam por 600ms. View transitions se transformam por 500ms. A consistência não é acidente. É o som do mesmo relógio marcando o tempo em toda parte.

Filtragem Sem Surpresa

Os cards da página inicial aprendem a se comportar como bons cidadãos. Seu primeiro dever é aparecer, depois ficar parados. Ordenação e filtragem são feitas com FLIP, não com surpresa. FLIP significa First, Last, Invert, Play: medir onde cada card começa, aplicar o filtro ou a ordenação, medir onde cada card termina e então animar o delta.

A implementação em static/js/filter.js captura os retângulos delimitadores iniciais de todos os cards visíveis, aplica a lógica de filtro para esconder ou mostrar cards e depois captura as posições finais. Cada card que se moveu recebe um transform em CSS para invertê-lo de volta à posição inicial, e então o transform é removido durante a duração da animação. O card parece deslizar suavemente da posição antiga para a nova.

const rects = new Map();
cards.forEach(card => {
  rects.set(card, card.getBoundingClientRect());
});

// Apply filter (hides/shows cards, layout reflows)
applyFilter();

cards.forEach(card => {
  const oldRect = rects.get(card);
  const newRect = card.getBoundingClientRect();
  const dx = oldRect.left - newRect.left;
  const dy = oldRect.top - newRect.top;

  card.style.transform = `translate(${dx}px, ${dy}px)`;
  requestAnimationFrame(() => {
    card.style.transition = 'transform 500ms var(--motion-easing)';
    card.style.transform = 'none';
  });
});

A grade nunca salta. Ela respira. Quando a navegação chega de outro lugar, a revelação inicial por keyframe é totalmente ignorada. O script detecta navegação verificando document.referrer para a mesma origem. Se o referrer corresponder, ele assume que você navegou de outra página do site, e desabilita imediatamente a animação de entrada ao adicionar disable-animation à grade de posts e definir animation: none em cada card. O resultado é silencioso: nada pisca, nada reinicia, nada se impõe depois do fato.

Troca de Idioma Como Sobreposição de Tradução

Uma das continuidades mais sutis do sistema é a forma como a troca de idioma parece funcionar. Quando você clica no alternador de idioma, não está saindo da página. Está traduzindo a superfície que já está lendo. A posição de scroll é preservada. O estado do filtro é preservado. A URL muda de /en/index.html para /pt/index.html, mas seu lugar no documento não muda.

Isso é tratado em transitions.js pela detecção de uma troca de idioma: a estrutura do caminho é idêntica, exceto pelo segmento en ou pt. Se esse padrão for detectado, a posição de scroll é capturada antes do fetch e restaurada depois da troca do DOM. Se você estava no meio de um artigo longo em inglês e mudar para português, continua no meio do mesmo artigo. O conteúdo muda, mas o espaço não.

O mesmo princípio se estende ao estado do filtro na página inicial. Se você filtrou posts por ano ou tag e trocar de idioma, o filtro permanece ativo. Os posts traduzidos aparecem, mas os critérios de seleção são preservados. É assim que um site bilíngue parece um único documento visto por duas lentes, e não dois sites separados.

Customização Sem Scaffolding

A customização respeita o grão do sistema. O ponto de entrada é configuração e texto, não scaffolding. Abra _source/config.py e você encontrará tudo o que pode mudar: caminho base, definições de idioma, metadados do site e strings da interface para os dois idiomas. Mude o caminho base e o build escreverá links que obedecem. Ajuste variáveis tipográficas em static/css/styles.css e toda a superfície se aperta ou se afrouxa de maneira uniforme.

O conjunto de idiomas é a única suposição embutida. O inglês é a fonte da verdade. O português é uma tradução fiel. Se um terceiro idioma for necessário, a regra se estende: adicione um segundo espelho, dê a ele um cache, mantenha os hashes honestos. O builder é um script que você consegue ler de uma vez só. Se você consegue lê-lo, consegue mudá-lo. Não há mágica. Não há camadas ocultas. O sistema é exatamente tão complexo quanto o problema que resolve.

Por Que Nenhum Framework

Recusar um framework aqui não é nostalgia. É alinhamento com o problema. Um blog é texto, algumas imagens e a sensação de que atravessá-lo deve parecer virar uma página sem perder a linha que você estava lendo. As capacidades nativas do navegador já são suficientes. Python já é suficiente. As dependências são escolhidas como ferramentas sobre uma bancada pequena: se uma ferramenta não deixa o corte mais limpo, ela fica na gaveta.

A ausência de maquinaria não é uma ideologia. É manutenção praticada com antecedência. Cada linha de código neste sistema é código que vou precisar ler de novo em seis meses ou seis anos. Cada dependência é uma incompatibilidade futura em potencial. Cada abstração é um imposto sobre a compreensão. Então o sistema continua pequeno. Continua legível. Continua mutável.

A View Transitions API é nativa do Chromium. Não exige polyfill, wrapper nem etapa de build. O navegador já sabe como capturar dois estados do DOM e animar entre eles. Meu trabalho é decidir quando isso deve acontecer e impedir que o resto da página interfira. O CSS que descreve o movimento é declarativo. Ele se lê como uma partitura de coreografia, não como um programa.

As bibliotecas frontmatter e markdown do Python fazem exatamente uma coisa cada uma: analisar YAML e converter Markdown em HTML. Fazem isso bem. Não tentam ser sistemas de build. Não tentam ser roteadores. Elas analisam texto. É tudo de que preciso.

A Filosofia do Movimento

A filosofia do movimento é a filosofia da atenção. Nada deve insistir. As transições respondem a três perguntas: de onde, para onde e como este estado se relaciona com o anterior. Mudanças de tema ondulam como uma nuvem passando sobre o sol. Um card se eleva alguns pixels quando você passa o mouse, depois volta ao repouso. A superfície nunca mente sobre o que se moveu ou por quê.

Quando algo precisa ser removido, ele se dissolve ao longo de 500 milissegundos. Quando algo precisa chegar, emerge de onde poderia plausivelmente estar. O card de post na página inicial se expande até virar o artigo completo. O título permanece no lugar e cresce. A data se ancora no canto. O layout respira para fora sem romper a continuidade. Quando você aperta voltar, o artigo se contrai e volta a ser card. O espaço é conservado.

A curva de easing é a mesma em toda parte: cubic-bezier(0.4, 0, 0.2, 1). Isso não é uma escolha aleatória. É o ease padrão do Material Design, escolhido porque parece natural à percepção humana. O movimento começa com intenção, acelera suavemente e desacelera ao se completar. Nada estala. Nada desliza em velocidade constante. Tudo se move como se tivesse peso.

O Contrato com o Leitor

Para quem quiser adaptá-lo, o contrato é pequeno e explícito. Markdown entra, HTML sai. Um tradutor que respeita código e voz. Uma camada de navegação que nunca bloqueia a renderização com trabalho que poderia ter feito antes. Estilos que codificam ritmo uma vez e o reutilizam em toda parte. Se você mudar uma coisa, deve parecer que mudou exatamente uma coisa, e o resto do sistema deve responder com calma proporcional.

A única esperteza está em saber quando parar. O sistema poderia fazer mais. Poderia ter um CMS. Poderia ter preview em tempo real. Poderia ter analytics, testes A/B e widgets de compartilhamento social. Não faz nenhuma dessas coisas porque nenhuma delas melhora a experiência de leitura. O objetivo não é construir uma plataforma. O objetivo é escrever e deixar que a escrita seja lida sem interferência.

O Que o Sistema Sabe

Isto não é uma demonstração. É a versão mais simples de um documento vivo que eu pude fazer sem mentir sobre o que a web já consegue fazer. Na maioria das noites, escrevo com as janelas abertas em Copacabana e o som da avenida se misturando ao mar. O sistema aprende esse ritmo e o mantém. As coisas fluem não porque são rápidas, mas porque não atrapalham a si mesmas.

O cache se lembra do que não mudou. O tradutor preserva o que não deve ser traduzido. A navegação mantém onde você estava olhando quando o idioma mudou. O sistema de movimento codifica quanto tempo as coisas devem levar e se recusa a discutir sobre isso. Cada decisão que o sistema toma está a serviço de uma única ideia: o documento é contínuo. Quando você se move por ele, o espaço se reorganiza, mas nunca se rompe.

É isso que significa um site estático parecer vivo.
