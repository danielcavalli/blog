CONSTRUINDO UM BLOG BILÍNGUE COM VIEW TRANSITIONS

24 de Janeiro de 2025 • 11 min de leitura
Desenvolvimento WebView Transitions APIGemini AISites Estáticos

Como este blog funciona: geração estática em Python, tradução automática via Gemini, View Transitions API para uma navegação fluida e uma filosofia de construção baseada em padrões web nativos.

Este blog é uma plataforma construída com geração de site estático em Python, tradução automática via Gemini AI e transições de página fluidas usando a View Transitions API. Sem frameworks, sem dependências pesadas. Apenas padrões web nativos fazendo o trabalho pesado. A arquitetura é simples o suficiente para ser compreendida em uma tarde, poderosa o bastante para parecer contínua durante o uso e pequena o suficiente para ser mantida sozinho.

O objetivo do design não era construir o motor de blog mais completo em funcionalidades. Era construir o mais silencioso. Um lugar onde a escrita aparece sem cerimônia, onde a navegação pareça uma reorganização do espaço em vez de carregar novas páginas e onde um segundo idioma surja de forma tão natural quanto o primeiro. Cada escolha técnica serve a esse propósito.

Geração Estática com Propósito

No coração de tudo está um script em Python que lê arquivos markdown de _source/posts e escreve HTML em dois diretórios de saída: /enpara inglês e/pt para português. O script usa frontmatter para analisar metadata YAML e markdown para converter o conteúdo do corpo. Não existe sistema de plugins, nem mercado de temas, nem abstrações sobre abstrações. O script compõe arquivos. Isso é tudo o que ele precisa fazer.

Cada arquivo markdown contém metadados iniciais com título, data, trecho e tags opcionais. O script percorre o diretório de posts, carrega cada arquivo, extrai a metadata, converte o markdown para HTML e o injeta em um template. O template não é um sistema à parte. É uma f-string do Python que sabe onde a navegação entra, onde o título do post fica e como o rodapé deve fechar. O HTML de saída é gravado no disco com nomes de arquivos determinísticos derivados do slug do post.

def generate_post_html(post, post_number, lang='en'):
    lang_dir = LANGUAGES[lang]['dir']

O build é determinístico. Execute-o duas vezes com a mesma entrada e você terá uma saída idêntica byte a byte. Isso torna o cache trivial e os rebuilds previsíveis. O único estado que persiste entre builds é o cache de tradução, que discutiremos em um momento.

Tradução como Conteúdo de Primeira Classe

Cada post é escrito em inglês. Cada post é traduzido automaticamente para o português brasileiro usando o modelo Gemini 2.0 Flash do Google. A tradução não é algo pensado depois. Ela faz parte do processo de build, armazenada em cache por hash de conteúdo e regenerada apenas quando o código-fonte em inglês muda.

O tradutor fica em _source/translator.py. Ele mantém um cache JSON em _cache/translation-cache.json, indexado pelo slug do post e pelo hash de conteúdo. Antes de solicitar uma tradução, o script verifica se o hash de conteúdo atual corresponde ao hash em cache. Se corresponder, a tradução em cache é usada. Caso contrário, uma nova tradução é solicitada via Gemini API e o cache é atualizado.

O prompt de tradução é cuidadoso. Ele instrui o modelo a localizar, não a traduzir literalmente. Termos técnicos que brasileiros usam em inglês (como framework, backend, API, deploy e commit) são preservados. Blocos de código não são tocados. A voz narrativa é adaptada para soar natural em português, mantendo o mesmo tom e ritmo do original.

def translate_post(self, slug: str, frontmatter: Dict, content: str):
    content_hash = self._calculate_content_hash(content)
    cached = self.cache.get_translation(slug, content_hash)
    if cached:
        return cached

    prompt = self._build_translation_prompt(frontmatter, content)
    response = self.model.generate_content(prompt)
    translated = self._parse_translation_response(response.text)

    self.cache.set_translation(slug, content_hash, translated)
    return translated

O resultado são dois sites completos e paralelos. Ambos são HTML estático. Ambos são autossuficientes. Ambos podem ser servidos a partir de uma CDN sem infraestrutura de backend. O site em inglês vive em /en, o site em português em /pt e o usuário pode fazer o switch entre eles com um único clique. O alternador de idioma não recarrega a página. Ele navega com uma View Transition que preserva a posição de rolagem e o estado do filtro, fazendo com que a troca pareça uma sobreposição em vez de um salto.

Navegação que se Lembra

O site usa a View Transitions API para animar a transição entre páginas. Esta API é exclusiva para Chromium, o que significa que o site foca no Chrome, Edge, Brave e outros navegadores baseados em Chromium. Não há polyfill. Não há animação de reserva. Se a API não estiver disponível, a navegação funciona normalmente sem transições. Esta é uma escolha deliberada: recursos de ponta para usuários em navegadores modernos e degradação suave para todos os outros.

O script que lida com a navegação é pequeno. Ele fica em static/js/transitions.js. Quando você clica em um link interno, o script intercepta o clique, busca o HTML da nova página, o transforma em um fragmento de documento e inicia uma View Transition. Dentro do callback da transição, o script troca o título do documento, as folhas de estilo, o conteúdo principal e a navegação, atualizando a URL com history.pushState. O navegador captura o estado anterior, executa o callback para atualizar o DOM, captura o novo estado e anima a diferença.

async function navigateTo(url) {
  const response = await fetch(url, { cache: 'no-cache' });
  const html = await response.text();
  const newDoc = new DOMParser().parseFromString(html, 'text/html');

  const transition = document.startViewTransition(() => {
    document.title = newDoc.title;
    const main = document.querySelector('main');
    const newMain = newDoc.querySelector('main');
    if (main && newMain) main.replaceWith(newMain);
    history.pushState(null, '', url);
  });

  await transition.finished;
}

Todo o comportamento da animação é definido no CSS usando pseudo-elementos ::view-transition-*. O JavaScript apenas dispara a API. O CSS define como o conteúdo antigo desaparece, como o novo aparece e como elementos compartilhados se transformam entre os estados. Os cards de posts na página inicial se expandem para artigos completos. Os títulos permanecem no lugar e crescem. O fundo muda suavemente. A sensação não é a de trocar de página, mas de reorganizar uma única superfície contínua.

A curva de atenuação (easing) é a mesma em todos os lugares: cubic-bezier(0.4, 0, 0.2, 1). As durações são compartilhadas entre todas as interações: 300ms para efeitos rápidos de hover, 500ms para transições principais, 600ms para mudanças de tema. Essas constantes vivem em propriedades personalizadas do CSS e se propagam por cada animação no site. A consistência não é imposta por um framework. Ela está codificada no próprio sistema de design.

Troca de Idioma como Tradução Espacial

Uma das continuidades sutis do sistema é como a troca de idioma é percebida. Quando você faz o switch do inglês para o português, você não está saindo da página. Você está traduzindo a superfície que já está lendo. A posição de rolagem é preservada. O estado do filtro na página inicial é preservado. A URL muda de /en/index.htmlpara/pt/index.html, mas o seu lugar no documento não muda.

Esse comportamento é detectado no script de navegação. Se a estrutura do caminho for idêntica, exceto pelo segmento en ou pt, a navegação é classificada como um switch de idioma. A posição de rolagem é capturada antes da busca e restaurada após a troca do DOM. Se você estava no meio de um artigo em inglês e troca para o português, você continua no meio do mesmo artigo em português. O conteúdo muda, mas o espaço não.

O mesmo princípio se aplica à filtragem na página inicial. Se você filtrou os posts por ano ou tag e trocar de idioma, o filtro permanece ativo. Os posts traduzidos aparecem com os mesmos critérios de seleção. Isso faz com que a experiência bilíngue pareça um documento único visto através de duas lentes, em vez de dois sites separados.

Filtragem com FLIP

A página inicial lista todos os posts com um painel de filtro para ano, mês e tags. A filtragem é instantânea. Quando você seleciona um ano, os posts que não correspondem desaparecem gradualmente e os posts restantes se reorganizam em uma grade mais justa. A reorganização é animada usando a técnica FLIP: First, Last, Invert, Play (Primeiro, Último, Inverter, Tocar).

O script captura a posição inicial de cada card visível. Ele aplica o filtro, que oculta os cards que não correspondem e faz com que o layout se reorganize. Ele captura a posição final de cada card que permanece visível. Para cada card, calcula a diferença entre a posição antiga e a nova, aplica um transform do CSS para inverter o card de volta à sua posição inicial e, em seguida, anima o transform de volta a zero.

const rects = new Map();
cards.forEach(card => {
  rects.set(card, card.getBoundingClientRect());
});

applyFilter();

cards.forEach(card => {
  const oldRect = rects.get(card);
  const newRect = card.getBoundingClientRect();
  const dx = oldRect.left - newRect.left;
  const dy = oldRect.top - newRect.top;

  card.style.transform = `translate(${dx}px, ${dy}px)`;
  requestAnimationFrame(() => {
    card.style.transition = 'transform 500ms var(--motion-easing)';
    card.style.transform = 'none';
  });
});

A grade nunca pula. Ela exala. Os cards se dissolvem e deslizam para novas posições no mesmo intervalo de 500ms. O resultado parece deliberado, não surpreendente. O layout é sempre legível. O movimento explica o que mudou.

Transições de Tema e Engenharia Contra Cintilação

O site suporta temas claro e escuro. O alternador é um botão simples na barra de navegação. Quando você clica nele, o tema se espalha por toda a página ao longo de 600 milissegundos. As cores de fundo mudam, as cores do texto se ajustam, as bordas suavizam ou ganham nitidez e as sombras são recalculadas. A transição é suave porque é definida uma única vez, no CSS, com uma única classe aplicada ao corpo.

Um desafio durante o desenvolvimento foi evitar a cintilação visual (flicker) quando as transições de tema se sobrepunham às View Transitions. Se ambos os sistemas estivessem animando os mesmos elementos simultaneamente, o resultado era um breve clarão. A solução foi remover classes CSS conflitantes antes de iniciar uma View Transition e evitar seletores universais que aplicam transições a todos os elementos de uma vez.

A transição de tema é controlada. Ela só se aplica quando a classe theme-transitioning está presente no corpo. Quando você alterna o tema, a classe é adicionada, o atributo de tema é atualizado e a classe é removida após a conclusão da transição. Isso garante que as mudanças de tema sejam animadas, mas que a navegação não dispare transições de tema redundantes.

Outra fonte de cintilação era a remoção atrasada de animações CSS nos cards de posts. Na página inicial, os cards aparecem com uma animação escalonada quando a página carrega pela primeira vez. Em navegações subsequentes, essa animação não deve rodar. O script detecta a navegação verificando document.referrer. Se o referenciador corresponder à origem do site, o script assume que o usuário navegou de outra página e desativa imediatamente a animação de introdução adicionando disable-animation à grade de posts. Os cards aparecem instantaneamente, sem piscar ou resetar.

Filosofia da Simplicidade

A arquitetura não é mínima por ser mínima. Ela é mínima porque o problema é simples. Um blog é texto, algumas imagens e a sensação de que percorrê-lo deve ser como virar uma página sem perder o seu lugar. As capacidades nativas do navegador são suficientes. Python é suficiente. As dependências são escolhidas com cuidado. Se uma ferramenta não torna o corte mais limpo, ela não entra na base de código.

A View Transitions API é nativa do Chromium. Não requer polyfill, nem wrapper, nem etapa de build. O navegador já sabe como capturar dois estados do DOM e animar entre eles. Meu trabalho é decidir quando isso deve acontecer e evitar que o resto da página interfira. O CSS que descreve o movimento é declarativo. Ele se lê como uma partitura de coreografia, não como um programa.

As bibliotecas frontmatter e markdown do Python fazem exatamente uma coisa cada: analisar YAML e converter markdown para HTML. Elas fazem isso bem. Elas não tentam ser sistemas de build ou roteadores. Elas processam texto. Isso é tudo o que eu preciso que elas façam. Todo o script de build tem menos de 600 linhas de Python. Você pode lê-lo de uma só vez. Se você pode ler, você pode mudar.

Manutenção Antecipada

Recusar um framework aqui não é nostalgia. É um alinhamento com o problema. Cada linha de código neste sistema é um código que terei que ler novamente em seis meses ou seis anos. Cada dependência é uma incompatibilidade futura em potencial. Cada abstração é um imposto sobre a compreensão. Por isso, o sistema permanece pequeno. Permanece legível. Permanece mutável.

A ausência de maquinário não é uma ideologia. É manutenção praticada com antecedência. Quando adiciono um novo post, escrevo o markdown e executo o script de build. A tradução acontece automaticamente. O HTML é gerado. O cache é atualizado. O site está pronto para o deploy. Não há servidor de hot-reloading. Não há configuração de webpack. Não há um package.json com 300 dependências. Há um script Python, um arquivo markdown e uma pasta de HTML estático.

O Contrato com o Leitor

Para aqueles que desejam adaptá-lo, o contrato é pequeno e explícito. Markdown entra, HTML sai. Um tradutor que respeita o código e a voz. Uma camada de navegação que nunca bloqueia a renderização com trabalho que poderia ter sido feito antes. Estilos que codificam o ritmo uma vez e o reutilizam em todos os lugares. Se você mudar uma coisa, deve sentir que mudou exatamente uma coisa, e o resto do sistema deve responder com calma proporcional.

A única esperteza está em saber quando parar. O sistema poderia fazer mais. Poderia ter um CMS. Poderia ter visualização em tempo real. Poderia ter analytics, testes A/B e widgets de compartilhamento social. Ele não faz nada disso porque nada disso melhora a experiência de leitura. O objetivo não é construir uma plataforma. O objetivo é escrever e permitir que a escrita seja lida sem interferências.

Isto não é uma demonstração. É a versão mais simples de um documento vivo que eu poderia fazer sem mentir sobre o que a web já é capaz de fazer. O cache lembra o que não mudou. O tradutor preserva o que não deve ser traduzido. A navegação mantém onde você estava olhando quando o idioma mudou. O sistema de movimento codifica quanto tempo as coisas devem levar e se recusa a discutir sobre isso. Cada decisão que o sistema toma é em serviço de uma única ideia: o documento é contínuo. Quando você se move através dele, o espaço se reorganiza, mas nunca se quebra.

É isso que significa um site estático parecer vivo.
