CONSTRUINDO UM BLOG BILÍNGUE COM VIEW TRANSITIONS

24 de Janeiro de 2025 • 11 min de leitura
Desenvolvimento WebView Transitions APIGemini 2.0 FlashSites Estáticos

Como este blog funciona: geração estática em Python, tradução automática com Gemini 2.0 Flash, View Transitions API para uma navegação contínua e uma filosofia de construir com padrões nativos da web.

Este blog é uma plataforma construída com geração de sites estáticos em Python, tradução automática com IA via Gemini e transições de página fluidas usando a View Transitions API. Sem frameworks, sem dependências pesadas. Só padrões nativos da web fazendo trabalho de verdade. A arquitetura é simples o bastante para ser entendida em uma tarde, poderosa o bastante para parecer contínua no uso e pequena o bastante para ser mantida por uma pessoa só.

O objetivo de design não era construir o sistema de blog mais completo em recursos. Era construir o mais silencioso. Um lugar onde a escrita aparece sem cerimônia, onde a navegação parece reorganizar o espaço em vez de carregar páginas novas e onde um segundo idioma surge com a mesma naturalidade do primeiro. Cada escolha técnica serve a esse objetivo.

Geração Estática com Propósito#Link to this section

No centro de tudo há um script em Python que lê arquivos markdown de _source/posts e escreve HTML em dois diretórios de saída: /en para inglês e /pt para português. O script usa frontmatter para fazer o parse dos metadados em YAML e markdown para converter o corpo do texto. Não existe sistema de plugins, nem marketplace de temas, nem abstrações empilhadas sobre abstrações. O script compõe arquivos. É só isso que ele precisa fazer.

Cada arquivo markdown contém front matter com título, data, excerpt e tags opcionais. O script percorre o diretório de posts, carrega cada arquivo, extrai os metadados, converte o markdown em HTML e injeta o resultado em um template. O template não é um sistema à parte. É uma f-string em Python que sabe onde entra a navegação, onde fica o título do post e como o rodapé deve terminar. O HTML de saída é escrito em disco com nomes de arquivo determinísticos derivados do slug do post.

def generate_post_html(post, post_number, lang='en'):
    lang_dir = LANGUAGES[lang]['dir']

O build é determinístico. Rode-o duas vezes com a mesma entrada e a saída será idêntica byte a byte. Isso torna o cache trivial e deixa previsível cada nova geração do site. O único estado que persiste entre builds é o cache de tradução, sobre o qual falo daqui a pouco.

Tradução como Conteúdo Central#Link to this section

Todo post é escrito em inglês. Todo post é traduzido automaticamente para o português do Brasil usando o modelo Gemini 2.0 Flash, do Google. A tradução não é um detalhe colocado no fim. Ela faz parte do processo de build, fica em cache por hash de conteúdo e só é regenerada quando a fonte em inglês muda.

O tradutor fica em _source/translator.py. Ele mantém um cache em JSON em _cache/translation-cache.json, indexado por slug do post e hash de conteúdo. Antes de pedir uma tradução, o script verifica se o hash atual do conteúdo corresponde ao hash armazenado no cache. Se corresponder, usa a tradução em cache. Se não, faz uma nova requisição via API do Gemini e atualiza o cache.

O prompt de tradução é cuidadoso. Ele orienta o modelo a localizar o texto, não a traduzir de forma literal. Termos técnicos que brasileiros costumam usar em inglês, como framework, backend, API, deploy e commit, são preservados. Blocos de código ficam intactos. A voz narrativa é adaptada para soar natural em português, mantendo o mesmo tom e o mesmo ritmo do original.

def translate_post(self, slug: str, frontmatter: Dict, content: str):
    content_hash = self._calculate_content_hash(content)
    cached = self.cache.get_translation(slug, content_hash)
    if cached:
        return cached

    prompt = self._build_translation_prompt(frontmatter, content)
    response = self.model.generate_content(prompt)
    translated = self._parse_translation_response(response.text)

    self.cache.set_translation(slug, content_hash, translated)
    return translated

O resultado são dois sites completos e paralelos. Ambos são HTML estático. Ambos são autocontidos. Ambos podem ser servidos por uma CDN sem infraestrutura de backend. O site em inglês vive em /en, o site em português em /pt, e o usuário pode alternar entre eles com um único clique. O botão de troca de idioma não recarrega a página. Ele navega com uma View Transition que preserva a posição de rolagem e o estado dos filtros, fazendo a mudança parecer uma sobreposição, não um salto.

Navegação que se Lembra#Link to this section

O site usa a View Transitions API para animar a navegação entre páginas. Essa API é exclusiva do Chromium, o que significa que o site é voltado para Chrome, Edge, Brave e outros navegadores baseados em Chromium. Não há polyfill. Não há animação de fallback. Se a API não estiver disponível, a navegação funciona normalmente, sem transições. É uma troca deliberada: recursos de ponta para quem usa navegadores modernos, degradação elegante para todo o resto.

O script que cuida da navegação é pequeno. Ele fica em static/js/transitions.js. Quando você clica em um link interno, o script intercepta o clique, busca o HTML da nova página, faz o parse em um documento e inicia uma View Transition. Dentro do callback da transição, o script troca o título do documento, as folhas de estilo, o conteúdo principal e a navegação, e depois atualiza a URL com history.pushState. O navegador captura o estado anterior, executa o callback para atualizar o DOM, captura o estado seguinte e anima a diferença.

async function navigateTo(url) {
  const response = await fetch(url, { cache: 'no-cache' });
  const html = await response.text();
  const newDoc = new DOMParser().parseFromString(html, 'text/html');

  const transition = document.startViewTransition(() => {
    document.title = newDoc.title;
    const main = document.querySelector('main');
    const newMain = newDoc.querySelector('main');
    if (main && newMain) main.replaceWith(newMain);
    history.pushState(null, '', url);
  });

  await transition.finished;
}

Todo o comportamento de animação é definido em CSS usando pseudo-elementos ::view-transition-*. O JavaScript só dispara a API. O CSS define como o conteúdo antigo desaparece, como o conteúdo novo entra e como elementos compartilhados se transformam entre um estado e outro. Os cards de post da página de índice se expandem até virar artigos completos. Os títulos permanecem no lugar e crescem. O fundo muda suavemente. A sensação não é de trocar de página, mas de reorganizar uma única superfície contínua.

A curva de easing é a mesma em todo o site: cubic-bezier(0.4, 0, 0.2, 1). As durações se repetem em todas as interações: 300ms para efeitos rápidos de hover, 500ms para as transições centrais, 600ms para mudanças de tema. Essas constantes ficam em propriedades customizadas de CSS e se propagam por toda animação do site. A consistência não é imposta por um framework. Ela está codificada no próprio sistema de design.

Troca de Idioma como Tradução Espacial#Link to this section

Uma das continuidades mais sutis do sistema está em como a troca de idioma acontece. Quando você alterna do inglês para o português, não está saindo da página. Está traduzindo a superfície que já estava lendo. A posição de rolagem é preservada. O estado dos filtros na página de índice é preservado. A URL muda de /en/index.html para /pt/index.html, mas o seu lugar no documento não.

Esse comportamento é detectado no script de navegação. Se a estrutura do caminho é idêntica, com exceção do segmento en ou pt, a navegação é classificada como troca de idioma. A posição de rolagem é capturada antes do fetch e restaurada depois da troca do DOM. Se você estava no meio de um artigo em inglês e muda para o português, continua no meio do mesmo artigo em português. O conteúdo muda, mas o espaço não.

O mesmo princípio vale para os filtros na página de índice. Se você filtrou os posts por ano ou tag e depois troca de idioma, o filtro continua ativo. Os posts traduzidos aparecem com os mesmos critérios de seleção. Isso faz a experiência bilíngue parecer um documento único visto por duas lentes, e não dois sites separados.

Filtragem com FLIP#Link to this section

A página de índice lista todos os posts com um painel de filtros por ano, mês e tags. A filtragem é instantânea. Quando você seleciona um ano, os posts que não correspondem desaparecem aos poucos e os que restam se reorganizam em uma grade mais compacta. Essa reorganização é animada com a técnica FLIP: First, Last, Invert, Play.

O script captura a posição inicial de cada card visível. Aplica o filtro, o que esconde os cards que não correspondem e faz o layout se reorganizar. Depois captura a posição final de cada card que continua visível. Para cada card, calcula a diferença entre a posição antiga e a nova, aplica um transform em CSS para inverter o card de volta à posição inicial e então anima esse transform de volta até zero.

const rects = new Map();
cards.forEach(card => {
  rects.set(card, card.getBoundingClientRect());
});

applyFilter();

cards.forEach(card => {
  const oldRect = rects.get(card);
  const newRect = card.getBoundingClientRect();
  const dx = oldRect.left - newRect.left;
  const dy = oldRect.top - newRect.top;

  card.style.transform = `translate(${dx}px, ${dy}px)`;
  requestAnimationFrame(() => {
    card.style.transition = 'transform 500ms var(--motion-easing)';
    card.style.transform = 'none';
  });
});

A grade não dá tranco. Ela respira. Os cards desaparecem aos poucos e deslizam para novas posições dentro da mesma janela de 500ms. O resultado parece deliberado, não surpreendente. O layout continua sempre legível. O movimento explica o que mudou.

Transições de Tema e Engenharia para Evitar Flicker#Link to this section

O site suporta temas claro e escuro. O controle é um botão simples na barra de navegação. Quando você clica nele, o tema se espalha pela página inteira ao longo de 600 milissegundos. As cores de fundo mudam, as cores do texto se ajustam, as bordas ficam mais suaves ou mais nítidas e as sombras se recalculam. A transição é fluida porque foi definida uma vez, em CSS, com uma única classe aplicada ao body.

Um dos desafios durante o desenvolvimento foi evitar flicker quando as transições de tema se sobrepunham às View Transitions. Se os dois sistemas animassem os mesmos elementos ao mesmo tempo, o resultado era um breve clarão. A solução foi remover classes CSS conflitantes antes de iniciar uma View Transition e evitar seletores universais que aplicam transições a todos os elementos de uma vez.

A transição de tema é controlada. Ela só se aplica quando a classe theme-transitioning está presente no body. Quando você alterna o tema, a classe é adicionada, o atributo de tema é atualizado e a classe é removida depois que a transição termina. Isso garante que mudanças de tema sejam animadas, mas que a navegação não dispare transições de tema redundantes.

Outra fonte de flicker era a remoção tardia das animações CSS nos cards de post. Na página de índice, os cards aparecem com uma animação escalonada de fade quando a página carrega pela primeira vez. Nas navegações seguintes, essa animação não deve rodar. O script detecta a navegação verificando document.referrer. Se o referrer corresponder à origem do site, o script assume que o usuário veio de outra página e desativa imediatamente a animação de entrada ao adicionar disable-animation à grade de posts. Os cards aparecem na hora, sem piscar nem reiniciar.

Filosofia da Simplicidade#Link to this section

A arquitetura não é minimalista por gosto. Ela é minimalista porque o problema é simples. Um blog é texto, algumas imagens, e a sensação de que navegar por ele deve ser como virar uma página sem perder o lugar. Os recursos nativos do navegador bastam. Python basta. As dependências são escolhidas com cuidado. Se uma ferramenta não deixa a solução mais limpa, ela não entra na base de código.

A View Transitions API é nativa do Chromium. Não exige polyfill, nem wrapper, nem etapa de build. O navegador já sabe capturar dois estados do DOM e animar a passagem entre eles. Meu trabalho é decidir quando isso deve acontecer e impedir que o resto da página interfira. O CSS que descreve o movimento é declarativo. Ele se lê como uma partitura de coreografia, não como um programa.

As bibliotecas frontmatter e markdown do Python fazem exatamente uma coisa cada uma: fazem o parse de YAML e convertem Markdown em HTML. Fazem isso bem. Não tentam virar sistemas de build nem ferramentas de roteamento. Fazem parse de texto. É só isso que eu preciso que façam. O script inteiro de build tem menos de 600 linhas de Python. Dá para ler de uma vez. Se você consegue ler, consegue mudar.

Manutenção Antecipada#Link to this section

Recusar um framework aqui não é nostalgia. É alinhamento com o problema. Cada linha de código neste sistema é uma linha que eu vou ter de reler em seis meses ou em seis anos. Cada dependência é uma incompatibilidade futura em potencial. Cada abstração é um imposto sobre a compreensão. Por isso o sistema continua pequeno. Continua legível. Continua fácil de mudar.

A ausência de maquinaria não é ideologia. É manutenção praticada de antemão. Quando adiciono um novo post, escrevo markdown e rodo o script de build. A tradução acontece automaticamente. O HTML é gerado. O cache é atualizado. O site está pronto para deploy. Não existe servidor com hot-reloading. Não existe configuração de webpack. Não existe package.json com 300 dependências. Existe um script em Python, um arquivo markdown e uma pasta de HTML estático.

O Contrato com o Leitor#Link to this section

Para quem quiser adaptar esse sistema, o contrato é pequeno e explícito. Entra Markdown, sai HTML. Um tradutor que respeita código e voz. Uma camada de navegação que nunca atrasa o paint com trabalho que poderia ter sido feito antes. Estilos que codificam o ritmo uma vez e o reutilizam em todo lugar. Se você muda uma coisa, a sensação deve ser a de que mudou exatamente uma coisa, e o resto do sistema deve responder com calma proporcional.

A única esperteza aqui é saber quando parar. O sistema poderia fazer mais. Poderia ter um CMS. Poderia ter preview em tempo real. Poderia ter analytics, testes A/B e widgets de compartilhamento social. Não faz nada disso porque nada disso melhora a experiência de leitura. O objetivo não é construir uma plataforma. O objetivo é escrever e deixar que a escrita seja lida sem interferência.

Isto não é uma demonstração. É a versão mais simples de um documento vivo que eu consegui fazer sem mentir sobre o que a web já é capaz de fazer. O cache se lembra do que não mudou. O tradutor preserva o que não deve ser traduzido. A navegação mantém o lugar onde você estava olhando quando o idioma muda. O sistema de movimento codifica quanto tempo as coisas devem levar e se recusa a discutir isso. Cada decisão que o sistema toma serve a uma única ideia: o documento é contínuo. Quando você se move por ele, o espaço se reorganiza, mas nunca se rompe.

É isso que faz um site estático parecer vivo.
