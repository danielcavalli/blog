CONSTRUINDO UM BLOG BILÍNGUE

24 de Janeiro de 2025 • 11 min de leitura

Este blog é uma plataforma construída com geração estática em Python, tradução automática via Gemini AI e transições de página fluidas usando a View Transitions API. Sem frameworks, sem dependências pesadas. Só padrões nativos da web fazendo trabalho de verdade. A arquitetura é simples o bastante para entender em uma tarde, poderosa o bastante para parecer contínua no uso e pequena o bastante para manter sozinho.

O objetivo do design não era construir o motor de blog mais completo em recursos. Era construir o mais silencioso. Um lugar em que a escrita aparece sem cerimônia, em que a navegação parece uma reorganização do espaço em vez do carregamento de novas páginas e em que uma segunda língua surge com a mesma naturalidade da primeira. Cada escolha técnica serve a esse objetivo.

Geração Estática com Propósito

No centro há um script em Python que lê arquivos markdown de _source/posts e grava HTML em dois diretórios de saída: /en para inglês e /pt para português. O script usa frontmatter para analisar metadados YAML e markdown para converter o conteúdo do corpo. Não há sistema de plugins, não há marketplace de temas, não há abstrações sobre abstrações. O script compõe arquivos. É só isso que ele precisa fazer.

Cada arquivo markdown contém front matter com título, data, resumo e tags opcionais. O script percorre o diretório de posts, carrega cada arquivo, extrai os metadados, converte o markdown para HTML e o injeta em um template. O template não é um sistema separado. É uma f-string em Python que sabe onde a navegação vai, onde o título do post fica e como o rodapé deve fechar. O HTML de saída é gravado em disco com nomes de arquivo deterministas derivados do slug do post.

def generate_post_html(post, post_number, lang='en'):
    lang_dir = LANGUAGES[lang]['dir']

A build é determinista. Se você executá-la duas vezes com a mesma entrada, obtém uma saída idêntica byte por byte. Isso torna o cache trivial e as rebuilds previsíveis. O único estado que persiste entre builds é o cache de tradução, do qual vamos falar em instantes.

Tradução como Conteúdo de Primeira Classe

Cada post é escrito em inglês. Cada post é traduzido automaticamente para português brasileiro usando o modelo Gemini 2.0 Flash do Google. A tradução não é um detalhe de última hora. Ela faz parte do processo de build, é armazenada em cache por hash de conteúdo e regenerada apenas quando a fonte em inglês muda.

O tradutor fica em _source/translator.py. Ele mantém um cache JSON em _cache/translation-cache.json, indexado por slug do post e hash de conteúdo. Antes de solicitar uma tradução, o script verifica se o hash atual do conteúdo corresponde ao hash em cache. Se corresponder, a tradução em cache é usada. Caso contrário, uma nova tradução é solicitada via Gemini API e o cache é atualizado.

O prompt de tradução é cuidadoso. Ele orienta o modelo a localizar o texto, não a traduzi-lo literalmente. Termos técnicos que brasileiros usam em inglês, como framework, backend, API, deploy e commit, são preservados. Blocos de código ficam intocados. A voz narrativa é adaptada para soar natural em português, mantendo o mesmo tom e ritmo do original.

def translate_post(self, slug: str, frontmatter: Dict, content: str):
    content_hash = self._calculate_content_hash(content)
    cached = self.cache.get_translation(slug, content_hash)
    if cached:
        return cached

    prompt = self._build_translation_prompt(frontmatter, content)
    response = self.model.generate_content(prompt)
    translated = self._parse_translation_response(response.text)

    self.cache.set_translation(slug, content_hash, translated)
    return translated

O resultado são dois sites completos e paralelos. Ambos são HTML estático. Ambos são autocontidos. Ambos podem ser servidos por uma CDN sem infraestrutura de backend. O site em inglês fica em /en, o site em português em /pt, e você pode alternar entre eles com um único clique. O alternador de idioma não recarrega a página. Ele navega com uma View Transition que preserva a posição de rolagem e o estado do filtro, fazendo a troca parecer uma sobreposição em vez de um salto.

Navegação que Lembra

O site usa a View Transitions API para animar entre páginas. Essa API é exclusiva do Chromium, o que significa que o site tem como alvo Chrome, Edge, Brave e outros navegadores baseados em Chromium. Não há polyfill. Não há animação de fallback. Se a API não estiver disponível, a navegação funciona normalmente sem transições. É uma escolha deliberada: recursos de ponta para usuários em navegadores modernos, degradação elegante para todo mundo.

O script que cuida da navegação é pequeno. Ele fica em static/js/transitions.js. Quando você clica em um link interno, o script intercepta o clique, busca o HTML da nova página, faz o parse para um fragmento de documento e inicia uma View Transition. Dentro do callback da transição, o script troca o título do documento, folhas de estilo, conteúdo principal e navegação, e depois atualiza a URL com history.pushState. O navegador captura o estado "antes", executa o callback para atualizar o DOM, captura o estado "depois" e anima a diferença.

async function navigateTo(url) {
  const response = await fetch(url, { cache: 'no-cache' });
  const html = await response.text();
  const newDoc = new DOMParser().parseFromString(html, 'text/html');

  const transition = document.startViewTransition(() => {
    document.title = newDoc.title;
    const main = document.querySelector('main');
    const newMain = newDoc.querySelector('main');
    if (main && newMain) main.replaceWith(newMain);
    history.pushState(null, '', url);
  });

  await transition.finished;
}

Todo o comportamento da animação é definido em CSS usando pseudo-elementos ::view-transition-*. O JavaScript apenas dispara a API. O CSS define como o conteúdo antigo desaparece, como o novo conteúdo aparece e como elementos compartilhados se transformam entre estados. Cartões de post no índice se expandem em artigos completos. Títulos permanecem no lugar e crescem. O fundo muda suavemente. A sensação não é de trocar de páginas, mas de reorganizar uma única superfície contínua.

A curva de easing é a mesma em toda parte: cubic-bezier(0.4, 0, 0.2, 1). As durações são compartilhadas por todas as interações: 300ms para efeitos rápidos de hover, 500ms para transições principais, 600ms para mudanças de tema. Essas constantes vivem em CSS custom properties e se propagam por toda animação no site. A consistência não é imposta por um framework. Ela está codificada no próprio sistema de design.

Troca de Idioma como Tradução Espacial

Uma das continuidades sutis do sistema é a sensação da troca de idioma. Quando você alterna de inglês para português, você não está saindo da página. Você está traduzindo a superfície que já está lendo. A posição de rolagem é preservada. O estado do filtro na página de índice é preservado. A URL muda de /en/index.html para /pt/index.html, mas o seu lugar no documento não.

Esse comportamento é detectado no script de navegação. Se a estrutura do caminho for idêntica exceto pelo segmento en ou pt, a navegação é classificada como troca de idioma. A posição de rolagem é capturada antes do fetch e restaurada depois da troca de DOM. Se você estava no meio de um artigo em inglês e muda para português, continua no meio do mesmo artigo em português. O conteúdo muda, mas o espaço não.

O mesmo princípio se aplica à filtragem na página de índice. Se você filtrou posts por ano ou tag e mudar de idioma, o filtro continua ativo. Os posts traduzidos aparecem com os mesmos critérios de seleção. Isso faz a experiência bilíngue parecer um único documento visto por duas lentes, em vez de dois sites separados.

Filtragem com FLIP

A página de índice lista todos os posts com um painel de filtros por ano, mês e tags. A filtragem é instantânea. Quando você seleciona um ano, os posts que não correspondem desaparecem e os restantes se reorganizam em uma grade mais compacta. Essa reorganização é animada usando a técnica FLIP: First, Last, Invert, Play.

O script captura a posição inicial de cada cartão visível. Ele aplica o filtro, que oculta cartões que não correspondem e faz o layout recalcular. Depois captura a posição final de cada cartão que continua visível. Para cada cartão, calcula a diferença entre a posição antiga e a nova, aplica um transform em CSS para inverter o cartão de volta à sua posição inicial e então anima esse transform de volta a zero.

const rects = new Map();
cards.forEach(card => {
  rects.set(card, card.getBoundingClientRect());
});

applyFilter();

cards.forEach(card => {
  const oldRect = rects.get(card);
  const newRect = card.getBoundingClientRect();
  const dx = oldRect.left - newRect.left;
  const dy = oldRect.top - newRect.top;

  card.style.transform = `translate(${dx}px, ${dy}px)`;
  requestAnimationFrame(() => {
    card.style.transition = 'transform 500ms var(--motion-easing)';
    card.style.transform = 'none';
  });
});

A grade nunca salta. Ela se acomoda. Os cartões se dissolvem e deslizam para novas posições na mesma janela de 500ms. O resultado parece deliberado, não surpreendente. O layout está sempre legível. O movimento explica o que mudou.

Transições de Tema e Engenharia Anti-Cintilação

O site suporta temas claro e escuro. O alternador é um botão simples na barra de navegação. Quando você clica nele, o tema se espalha por toda a página ao longo de 600 milissegundos. Cores de fundo mudam, cores do texto se ajustam, bordas ficam mais suaves ou mais nítidas e sombras são recalculadas. A transição é suave porque está definida uma vez, em CSS, com uma única classe aplicada ao body.

Um desafio durante o desenvolvimento foi evitar cintilação visual quando transições de tema se sobrepunham a View Transitions. Se ambos os sistemas estivessem animando os mesmos elementos ao mesmo tempo, o resultado era um breve flash. A solução foi remover classes CSS conflitantes antes de iniciar uma View Transition e evitar seletores universais que aplicam transições a todos os elementos de uma vez.

A transição de tema é controlada. Ela só se aplica quando a classe theme-transitioning está presente no body. Quando você alterna o tema, a classe é adicionada, o atributo de tema é atualizado e a classe é removida depois que a transição termina. Isso garante que mudanças de tema sejam animadas, mas a navegação não dispare transições de tema redundantes.

Outra fonte de cintilação era a remoção tardia de animações CSS nos cartões de post. Na página de índice, os cartões aparecem com um fade-in escalonado quando a página carrega pela primeira vez. Em navegações posteriores, essa animação não deve rodar. O script detecta a navegação verificando document.referrer. Se o referrer corresponder à origem do site, o script assume que você navegou de outra página e desabilita imediatamente a animação de entrada ao adicionar disable-animation à grade de posts. Os cartões aparecem instantaneamente sem piscar nem reiniciar.

Filosofia da Simplicidade

A arquitetura não é mínima pelo prazer de ser mínima. Ela é mínima porque o problema é simples. Um blog é texto, algumas imagens e a sensação de que navegar por ele deve parecer virar uma página sem perder o lugar. As capacidades nativas do navegador bastam. Python basta. As dependências são escolhidas com cuidado. Se uma ferramenta não simplifica de fato o sistema, ela não entra no codebase.

A View Transitions API é nativa do Chromium. Ela não exige polyfill, wrapper nem etapa de build. O navegador já sabe como capturar dois estados do DOM e animar entre eles. Meu trabalho é decidir quando isso deve acontecer e impedir que o resto da página interfira. O CSS que descreve o movimento é declarativo. Ele se lê como uma partitura de coreografia, não como um programa.

As bibliotecas frontmatter e markdown de Python fazem exatamente uma coisa cada: analisar YAML e converter Markdown em HTML. Elas fazem isso bem. Não tentam ser sistemas de build nem roteadores. Elas analisam texto. É só isso que eu preciso que façam. O script inteiro de build tem menos de 600 linhas de Python. Você consegue lê-lo de uma vez só. Se consegue lê-lo, consegue mudá-lo.

Manutenção com Antecedência

Recusar um framework aqui não é nostalgia. É alinhamento com o problema. Cada linha de código nesse sistema é código que vou ter de ler de novo em seis meses ou seis anos. Cada dependência é uma potencial incompatibilidade futura. Cada abstração é um imposto sobre a compreensão. Então o sistema continua pequeno. Continua legível. Continua maleável.

A ausência de maquinaria não é uma ideologia. É manutenção praticada com antecedência. Quando adiciono um novo post, escrevo markdown e executo o script de build. A tradução acontece automaticamente. O HTML é gerado. O cache é atualizado. O site está pronto para deploy. Não há servidor com hot reload. Não há configuração de webpack. Não há package.json com 300 dependências. Há um script em Python, um arquivo markdown e uma pasta de HTML estático.

O Contrato com o Leitor

Para quem quiser adaptar isso, o contrato é pequeno e explícito. Markdown entra, HTML sai. Um tradutor que respeita código e voz. Uma camada de navegação que nunca bloqueia a renderização com trabalho que poderia ter feito antes. Estilos que codificam o ritmo uma vez e o reutilizam em toda parte. Se você mudar uma coisa, a sensação deve ser a de que mudou exatamente uma coisa, e o resto do sistema deve responder com calma proporcional.

A única esperteza está em saber quando parar. O sistema poderia fazer mais. Poderia ter um CMS. Poderia ter preview em tempo real. Poderia ter analytics, testes A/B e widgets de compartilhamento social. Não faz nenhuma dessas coisas porque nenhuma delas melhora a experiência de leitura. O objetivo não é construir uma plataforma. O objetivo é escrever, e deixar que a escrita seja lida sem interferência.

Isto não é uma demonstração. É a versão mais simples de um documento vivo que eu consegui fazer sem mentir sobre o que a web já consegue fazer. O cache lembra do que não mudou. O tradutor preserva o que não deve ser traduzido. A navegação mantém onde você estava olhando quando o idioma muda. O sistema de movimento codifica quanto tempo as coisas devem levar e se recusa a discutir sobre isso. Cada decisão que o sistema toma está a serviço de uma única ideia: o documento é contínuo. Quando você se move por ele, o espaço se reorganiza, mas nunca se rompe.

É isso que significa um site estático parecer vivo.
