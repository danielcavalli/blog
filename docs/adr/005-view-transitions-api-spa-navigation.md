# ADR 5: View Transitions API for SPA Navigation

## Context

The dan.rio blog is designed around the principle that the site is "one continuous surface that reorganizes itself" rather than a collection of discrete pages. The design philosophy, documented in .github/design-philosophy.md, demands that every navigation feels like the environment reconfiguring — elements arrive and depart rather than appear and vanish. This rules out traditional multi-page architecture where a full page reload interrupts the spatial illusion with a white flash and a loss of animation context.

The site targets Chromium browsers exclusively, as documented in the project instructions and `.github` design documents. This constraint was accepted early and means the project can depend on APIs that only Chrome, Edge, and Brave implement without polyfill or fallback complexity. The View Transitions API, shipped in Chrome 111, is one such API. It allows a developer to call `document.startViewTransition()` with a callback that mutates the DOM; the browser automatically captures the visual state before and after the callback and animates between the two snapshots using CSS pseudo-elements.

The site has four page types — a root landing page, bilingual index pages with filterable post grids, individual post pages, and an about/CV page — served as static HTML from GitHub Pages. There is no server-side rendering or client-side framework. Navigation between these pages requires either full page loads, a framework router that manages a component tree, or a custom mechanism that fetches HTML and swaps the DOM while the browser handles the visual transition.

The filtering system on the index page is not a simple show/hide. It implements a three-phase choreography — dissolve non-matching cards, reorganize remaining cards to fill gaps using the FLIP technique, then reveal newly matching cards with staggered entrances. This choreography runs inside filter.js, which spans 985 lines and manages its own animation state through CSS classes and custom properties. The FLIP technique (First-Last-Invert-Play) captures element positions before a layout change, lets the browser reflow, then applies inverse CSS transforms and animates them to zero. Both the page transition system and the filter animation system must coexist on the same DOM without interfering with each other's timing or state.

After any SPA-style DOM swap, scripts that bound event listeners to the replaced elements lose their references. The site has no component lifecycle system to handle this. Each script needs a reliable signal that the DOM has changed so it can rebind.

## Decision

We will use the Chromium-native View Transitions API as the sole mechanism for SPA-like page navigation, implemented in transitions.js. The file intercepts internal link clicks via a document-level click listener, fetches the target page as HTML using the Fetch API with cache-busting headers, parses it with DOMParser, then calls document.startViewTransition() with a synchronous callback that swaps the document title, stylesheets, main content, nav link states, lang attribute, and history entry. The browser captures the old and new visual states and animates between them using CSS-defined transitions on ::view-transition-* pseudo-elements.

We will define all transition choreography in CSS rather than JavaScript. The first 100 lines of styles.css declare view transition behavior: the root transition is hidden to prevent double-rendering, named groups like site-nav and site-footer receive their own animation rules, and the blog-header slides up behind the nav bar when navigating to a post and slides back down when returning. Global timing is set via CSS custom properties (--motion-duration-core at 500ms, --motion-easing as a cubic-bezier curve). JavaScript never calls requestAnimationFrame to animate view transition elements; it only calls startViewTransition() and lets CSS handle the rest.

We will use the FLIP technique in filter.js for spatial reorganization of post cards during filtering and sorting. The implementation captures bounding rectangles before a layout change (First), applies the change (Last), calculates positional deltas and sets inverse CSS transforms via --flip-x and --flip-y custom properties (Invert), then lets CSS transitions animate the transforms to zero (Play). This runs independently of the View Transitions API and operates on the live DOM after page transitions have completed.

We will use a custom event, page-navigation-complete, as the coordination signal between the navigation system and all other scripts. After a view transition finishes and the DOM is settled, transitions.js dispatches this event on document. Scripts like filter.js and theme.js listen for it and reinitialize their event listeners on the new DOM elements. The landing page morph in landing.js uses the same pattern — after its own startViewTransition() completes and newly injected scripts have loaded, it dispatches the same event. Scripts use data-*-initialized attributes as guards against duplicate initialization.

We will preserve continuity across language switches by detecting when the navigation path changes only in its /en/ or /pt/ prefix. When a language switch is detected, transitions.js captures the current scroll position and filter state before the swap, restores scroll position after the swap so the reader stays in the same place, and restores filter selections (year, month, tags, sort order) via synthetic clicks after filter.js reinitializes. Regular navigation between different pages resets scroll to top.

We will handle browser history manually. transitions.js sets history.scrollRestoration to manual, stamps each history entry with its scrollY position via replaceState before pushing a new entry, and restores the saved position on popstate. The navigateTo function accepts an isPop flag to distinguish forward navigation (which calls pushState) from back/forward navigation (which skips pushState since the browser has already updated the URL). An AbortController cancels in-flight fetches when a new navigation begins before the previous one completes.

## Status

Accepted.

## Consequences

The animation system is split cleanly between CSS and JavaScript. CSS owns all visual choreography for page transitions via ::view-transition-* pseudo-elements and for filter animations via transition properties on card classes. JavaScript owns timing coordination, DOM manipulation, and state management but never directly animates elements during view transitions. This separation means a designer can change transition durations, easing curves, or animation directions entirely in CSS without touching JavaScript.

The site carries zero animation library dependencies. There is no GSAP, no Framer Motion, no anime.js, no barba.js. The entire animation system is four hand-written JavaScript files and CSS custom properties. The browser's compositor handles the view transition rendering, which means animations run at 60fps even during the DOM swap because the browser is animating cached snapshots, not live elements.

Progressive enhancement works naturally. The first line of transitions.js checks for document.startViewTransition support and returns immediately if absent. Without JavaScript or on non-Chromium browsers, every link is a standard anchor tag pointing to a real static HTML file. The site degrades to a traditional multi-page site with no animation rather than breaking.

The Chromium-only constraint is the most significant tradeoff. Firefox and Safari users get the static multi-page experience. As of this writing, Firefox has View Transitions behind a flag and Safari has no implementation. This was an intentional decision documented in copilot-instructions.md — the site prioritizes depth of experience on one engine over breadth across all browsers. A framework router like React Router, SvelteKit, or even a lighter library like Turbo or barba.js could provide cross-browser animated transitions, but each would introduce a runtime dependency and a component model that conflicts with the no-framework philosophy established in ADR 1.

The reinitialization system is inherently fragile. Every script that binds event listeners to elements inside main must listen for page-navigation-complete and rebind. If a new script is added and the author forgets to handle this event, it will work on initial page load but silently break after SPA navigation. The data-*-initialized guard pattern prevents double-initialization but adds a convention that must be remembered rather than enforced by a compiler or framework. A framework with a component lifecycle (mount, unmount, remount) would handle this automatically.

Manual scroll restoration adds complexity that browser-native MPA navigation and framework routers handle transparently. transitions.js maintains scrollY in history.state, distinguishes between forward navigation, back/forward navigation, and language switches, and restores scroll position at three different points depending on the navigation type. This logic, combined with the AbortController for canceling in-flight fetches and the landing page guard that forces a full reload for the root URL, accounts for a substantial portion of the 586 lines in transitions.js.

The FLIP implementation in filter.js is roughly 200 lines of manual position capture, delta calculation, transform application, and cleanup — functionality that GSAP's Flip plugin or the proposed CSS view-transitions-for-same-document specification would reduce to a few declarations. The implementation works reliably but must be maintained by hand as the card layout evolves. The three-phase choreography (dissolve at 400ms, reorganize via FLIP at 500ms, reveal with 60ms stagger) uses nested requestAnimationFrame calls and setTimeout chains that are correct but difficult to reason about when debugging timing issues.

The alternative approaches considered were traditional MPA (rejected because it breaks the continuous surface principle), React Router or SvelteKit (rejected per ADR 1's no-framework constraint), Turbo from Hotwire (rejected because its frame-based approach does not integrate with the View Transitions API's snapshot mechanism), and barba.js (rejected because it would be the only third-party JavaScript dependency and provides less control over the transition callback than calling startViewTransition() directly). The chosen approach trades library convenience for direct access to the platform primitive, consistent with the project's philosophy that working with native APIs is both a technical choice and a learning exercise.
