# ADR 10: Testing and CI Strategy

## Context

The dan.rio blog is built by a custom Python static site generator that depends on the Google Gemini API for its bilingual translation pipeline. A full site build parses Markdown posts, sends content through a multi-agent Gemini translation pipeline, generates HTML for both languages, and writes a sitemap and root landing page. This build requires a GEMINI_API_KEY environment variable and a populated translation cache in the git-ignored _cache directory. Neither of these is available in a typical CI environment, and providing an API key to CI would introduce cost, rate-limiting delays, and a secret management burden for a personal blog.

The build system's Python code falls into two distinct categories. One category is pure functions with no side effects: tag slug generation, reading time calculation, date formatting for two locales, SHA-256 content hashing, language path construction, and alternate language resolution. These functions live primarily in helpers.py and the utility layer of build.py. They accept simple inputs, return deterministic outputs, and have no dependencies beyond the standard library. The other category is orchestration code that calls the Gemini API, reads and writes files, manages the translation cache, and assembles full HTML pages from multiple data sources. This orchestration code cannot run without the full project environment.

Between the pure functions and the full build sits a middle layer of rendering functions that generate HTML fragments: navigation bars, footers, language toggles, skip links, post cards, JSON-LD structured data scripts, and theme toggle SVGs. These functions depend on configuration constants and bilingual string dictionaries from config.py but do not touch the filesystem or any external API. They are deterministic given their inputs and amenable to unit testing, though their output is raw HTML strings rather than parsed DOM structures.

The project uses uv as its package manager with pytest declared as a dev dependency in pyproject.toml. There is no linter, formatter, type checker, or pre-commit hook configured. The project philosophy explicitly avoids heavy tooling, and the JavaScript and CSS assets are hand-written vanilla code with no build step of their own.

## Historical Decision

We will use pytest as the sole test framework, with unit tests in tests/test_build.py covering pure helper functions and HTML rendering functions. The test file stubs out heavy external dependencies at the module level before importing the build module. It creates synthetic module objects for google, google.genai, and dotenv using types.ModuleType and injects them into sys.modules, then stubs translator.MultiAgentTranslator and translator.validate_translation with unittest.mock.MagicMock instances. This allows the entire build module to be imported and its pure functions exercised without a Gemini API key, without the google-genai package functioning, and without any network access.

We will test pure helper functions thoroughly with edge cases. The test suite covers tag_to_slug with inputs including mixed case, special characters, Unicode, empty strings, and leading whitespace. It covers calculate_reading_time with boundary conditions at zero, one, two hundred, and one thousand words. It covers format_date and format_iso_date for both English and Portuguese locales, including invalid input handling. It covers calculate_content_hash for determinism, uniqueness, and output format. It covers get_lang_path and get_alternate_lang for both language directions and unknown language fallback.

We will test HTML rendering functions by asserting on string contents rather than DOM structure. Tests for render_nav verify that the active page receives the correct CSS class and that Portuguese labels appear in the Portuguese variant. Tests for render_footer check for the presence of copyright text in both languages and ARIA-labeled social links. Tests for generate_post_card verify title uppercasing, view transition name generation, tag pill rendering, data attributes, locale-specific date formatting, link path construction, and HTML entity escaping. Tests for render_jsonld_script verify JSON roundtrip integrity, script tag escaping, and non-ASCII character preservation. Tests for generate_lang_toggle_html verify bidirectional linking, active class assignment, ARIA labels, and data attributes.

We will run a GitHub Actions CI pipeline defined in .github/workflows/validate.yml on every push to main and on every pull request. The pipeline installs dependencies with uv sync, runs py_compile syntax checks against all nine Python source files in the _source directory, and then runs the pytest suite. The pipeline does not attempt a full site build, does not require any secrets, and does not generate or deploy any HTML output.

## Consequences

The CI pipeline runs in seconds without any API credentials, network calls, or translation cache. Every push to main and every pull request verifies that all Python source files parse without syntax errors and that the pure function layer behaves correctly. This fast feedback loop catches regressions in date formatting, slug generation, reading time calculation, and HTML fragment rendering before they reach production. The module-level stubbing approach means tests can run on any machine with Python and pytest, including contributor environments that lack a Gemini API key.

The test suite provides strong coverage of the pure function boundary but does not test the build orchestration at all. No test ever calls the main build function, writes HTML files to disk, or verifies that a complete page assembles correctly from its component fragments. A regression in how the build orchestrator wires together the content loader, translator, renderer, and SEO module would not be caught until a manual local build. The gap between what is tested and what is deployed is significant: the CI pipeline validates individual bricks but never assembles or inspects the wall.

There are no integration tests that execute a full build against fixture content with a mocked translation layer. Such tests would verify that Markdown posts produce correct HTML pages with proper navigation, language switching, and SEO metadata, without requiring an API key. There are no end-to-end tests using a browser automation tool like Playwright or Cypress that would verify client-side behavior: View Transitions API navigation, theme toggling, post filtering animations, and landing page morphing. There are no visual regression tests that would catch unintended CSS changes by comparing screenshots across builds. There is no link checking to verify that internal links between the English and Portuguese versions resolve correctly. Any of these layers could regress silently.

There is no linting or formatting enforcement. No Ruff, ESLint, or Prettier runs in CI or locally. No pre-commit hooks gate what enters the repository. Code style is maintained by convention and author discipline alone. This is consistent with the project's philosophy of minimal tooling, but it means that style drift, unused imports, and minor errors in the JavaScript and CSS assets go unchecked by automation. The testing strategy reflects a deliberate tradeoff: fast, secret-free CI that validates the deterministic core, in exchange for manual verification of everything above the unit boundary.
