# ADR 9: SEO and Structured Data Strategy

## Context

The dan.rio blog is a bilingual personal site published in English and Brazilian Portuguese, hosted as static HTML on GitHub Pages. It has no server-side processing, no JavaScript framework, and no CMS. Every page is generated at build time by a Python static site generator and committed directly to the repository. The site competes for search engine visibility in two languages against platforms with far more authority, and every post is shared on social media where a rich preview card is the difference between a click and a scroll-past.

A bilingual static site creates specific SEO problems that a monolingual site does not face. Without explicit language signals, search engines may index the Portuguese version for English queries or vice versa, dilute ranking authority across duplicate-looking pages, or fail to surface the correct language variant for a user's locale. Google's documentation is clear that hreflang annotations are the mechanism for declaring language relationships between equivalent pages, and that these annotations must be reciprocal and consistent across all variants.

Static sites also have a caching problem. Because GitHub Pages serves files with aggressive cache headers and there is no server to set ETags or content negotiation headers, browsers and CDN edges may serve stale CSS and JavaScript long after a deployment. The site needs a cache-busting mechanism that is deterministic, so that unchanged assets keep the same URL across builds and only actually-modified files invalidate caches.

The site's author is a machine learning engineer whose name appears in several forms across the web: Daniel Cavalli, Dan Cavalli, Dan Rio. Search engines and AI systems build knowledge graphs from structured data, and without explicit identity signals, these entities may not be connected. The site needs to establish a single canonical identity that links these names to one person and to the correct social profiles.

## Decision

We will embed JSON-LD structured data on every page of the site, using schema.org vocabulary to describe the content and its author. The root landing page at dan.rio carries a WebSite object paired with a Person object that includes alternateName entries for all known name variants, sameAs links to GitHub, LinkedIn, and Twitter profiles, and knowsAbout entries for the author's professional domains. Blog post pages carry a BlogPosting object with headline, description, datePublished, dateModified, inLanguage, and keywords fields, with the author reference pointing to the same Person entity. The about page and CV page each carry a ProfilePage object whose mainEntity is the Person, enriched with a description field drawn from locale-specific configuration. All JSON-LD generation is centralized in seo.py, which exports render_person_jsonld as a reusable Person dictionary and render_jsonld_script as a serializer that produces a script tag with proper escaping, including protection against premature script-close sequences in content.

We will include Open Graph and Twitter Card meta tags on every page. Each page generator in renderer.py assembles an extra_meta string containing og:type, og:url, og:title, og:description, og:site_name, and og:locale tags, plus twitter:card, twitter:title, and twitter:description tags. Blog posts additionally include article:published_time, article:modified_time, and article:author properties. Profile pages include profile:first_name and profile:last_name. The og:locale:alternate tag is set on every bilingual page to signal the existence of the other language variant. These meta tags are passed into the shared render_head function in renderer.py, which places them alongside the canonical URL, author meta tag, and robots directive.

We will generate a sitemap.xml with full hreflang annotations for every URL on the site. The generate_sitemap function in seo.py produces a sitemap where every URL entry includes xhtml:link elements for x-default, English, and Portuguese variants. The x-default hreflang always points to the root landing page at dan.rio, which serves as the language-neutral entry point. The sitemap includes the root landing page at priority 1.0, static pages like index, about, and CV at priorities 0.7 through 0.9, and all blog posts at priority 0.8. Each entry carries a lastmod date derived from the post's frontmatter updated field when available, falling back to the published date. The robots.txt file allows all crawlers unconditionally and declares the sitemap location.

We will apply content-hash cache busting to all CSS and JavaScript URLs. The _asset_hash function in helpers.py computes the first eight characters of the SHA-256 hash of each asset file's contents and appends it as a query parameter. The render_head function in renderer.py calls _asset_hash for every stylesheet and script reference, producing URLs like styles.css?v=a1b2c3d4. Results are cached in a module-level dictionary for the lifetime of the build process. Because the hash is derived from file contents rather than build timestamps, unchanged assets produce identical URLs across builds, preserving browser caches, while any content change produces a new hash that forces a fresh download.

We will set a canonical URL on every page via a link rel="canonical" tag in render_head, and include reciprocal hreflang alternate links in the HTML head of every bilingual page, mirroring the annotations in the sitemap.

## Status

Accepted.

## Consequences

The JSON-LD structured data gives search engines and AI systems explicit signals about the site's content types and authorship. The Person entity with alternateName and sameAs fields enables knowledge graph disambiguation, connecting the author's various name forms and social profiles into a single entity. BlogPosting markup with datePublished and dateModified fields allows search engines to display rich results with publication dates and author attribution. The WebSite object on the landing page establishes the site as a named entity in its own right.

Open Graph and Twitter Card tags ensure that every page shared on social media produces a rich preview with title, description, and site name rather than a bare URL or an auto-extracted snippet. The og:locale and og:locale:alternate tags signal to social platforms that the content exists in multiple languages, which is particularly relevant for platforms that may choose which variant to display based on the viewer's locale.

The hreflang sitemap and HTML alternate links solve the bilingual indexing problem. Search engines can confidently serve the English version to English-language queries and the Portuguese version to Portuguese-language queries without treating the two as duplicate content. The x-default annotation pointing to the root landing page provides a fallback for locales that match neither language. Reciprocal annotations in both the sitemap and the HTML head reinforce each other, following Google's recommendation for redundant language signals.

Content-hash cache busting eliminates stale asset delivery without sacrificing cache efficiency. An asset that has not changed between deployments keeps the same query string and remains cached. An asset that has changed gets a new hash and is fetched fresh. This is deterministic and reproducible: two builds from the same source tree produce identical hashes, which means generated HTML diffs stay clean when only content changes.

There are real costs to this approach. The structured data is generated as Python dictionaries serialized to JSON inside f-strings, with no schema validation at build time. If a field is missing, misspelled, or has the wrong type, the JSON-LD will be technically valid JSON but semantically incorrect schema.org, and no part of the build will catch it. Google's Rich Results Test and Schema Markup Validator can check the output, but these are manual, external tools that are not integrated into the build pipeline. Adding a JSON Schema or a schema.org validator as a build step would catch these errors automatically but would introduce a new dependency.

The Open Graph and Twitter Card meta tag patterns are duplicated across four page generator functions in renderer.py: generate_post_html, generate_index_html, generate_about_html, and generate_cv_html. Each function assembles its own extra_meta string with slight variations. This repetition means that adding a new meta tag or changing the pattern requires edits in four places, which is a maintenance risk that could be resolved by extracting a shared helper function, at the cost of adding another layer of abstraction to the rendering pipeline.

There is no automated testing of SEO output. The build system has unit tests for pure helper functions like date formatting and tag slugification, but no tests verify that the generated HTML contains valid JSON-LD, correct hreflang annotations, or complete Open Graph tags. A regression in the structured data or meta tags would be invisible until manually checked or until search engine behavior changed. The alternative approaches considered were microdata, which embeds structured data directly in HTML attributes and is harder to maintain alongside f-string templates; RDFa, which has similar HTML-interleaving problems and lower tooling support; and omitting structured data entirely, which would forfeit rich results, knowledge graph presence, and proper bilingual indexing. JSON-LD was chosen because it is a self-contained script block that does not interfere with the HTML structure, can be generated independently from the page markup, and is Google's explicitly preferred format.
