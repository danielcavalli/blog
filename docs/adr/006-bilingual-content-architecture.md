# ADR 6: Bilingual Content Architecture

## Context

The dan.rio blog serves a dual audience. Its author, a Brazilian ML engineer based in Rio de Janeiro, writes naturally in English about technical topics but publishes to a readership that is substantially Brazilian Portuguese-speaking. Every post, the About page, and the CV need to appear in both languages for the site to fulfill its purpose. The question is not whether to support two languages but how to get Portuguese content into the system without doubling the authoring burden or compromising the integrity of the source files.

There are several established approaches to bilingual content. Manual bilingual authoring, where the author writes or maintains two copies of every document, provides full editorial control but creates an ongoing maintenance obligation that scales linearly with content volume. CMS-based internationalization systems like WordPress with WPML or Contentful's localization features provide UI-driven workflows but introduce infrastructure dependencies incompatible with the project's static-file, no-framework philosophy (see ADR 1). Translation management platforms such as Crowdin and Lokalise offer professional workflows with translator assignment, review stages, and translation memory, but they are designed for teams and introduce external service dependencies, webhook integrations, and per-seat costs that are disproportionate for a single-author blog. The project needs a solution that keeps the authoring surface minimal, automates the mechanical work, and never risks corrupting the source Markdown files that are the single editorial truth of the system.

A critical constraint is content integrity. The build system must treat source Markdown files as read-only inputs. Earlier iterations of the build stored derived metadata such as content hashes and timestamps directly in YAML frontmatter, which meant the build would write back into the files it was reading. This created a class of bugs where a build failure could leave frontmatter in an inconsistent state, and it muddied the git history with automated metadata churn alongside genuine editorial changes. Any bilingual architecture must respect the principle that the build reads from source files and writes only to output directories and cache artifacts.

## Decision

We will author all content exclusively in English Markdown with YAML frontmatter. The frontmatter carries only editorial fields that the author controls: title, date, excerpt, tags, and optionally slug, canonical, and updated. The build will never write to these source files.

We will translate content to Brazilian Portuguese at build time using the Gemini API, implemented as a multi-agent pipeline in translator.py. The build orchestrator in build.py parses each English Markdown post via parse_markdown_post in content_loader.py, then passes the raw Markdown and frontmatter to the MultiAgentTranslator, which handles the API interaction, response parsing, and quality validation. The translator produces Portuguese versions of the title, excerpt, tags, and body content. The same pipeline translates the About page content defined in the LANGUAGES dictionary in config.py and the structured CV data loaded from cv_data.yaml.

We will use SHA-256 content hashing for cache invalidation across two independent caching layers. The first layer is the sidecar metadata manifest at _cache/post-metadata.json, managed by content_loader.py. This manifest stores the content hash, first-seen timestamp, and last-change timestamp for each post slug, keyed by the SHA-256 digest computed by calculate_content_hash in helpers.py. When a post's content hash matches the stored value, the build knows the content has not changed since the last run. The second layer is the translation cache at _cache/translation-cache.json, managed by the TranslationCache class in translator.py. Each entry stores the SHA-256 hash alongside the translated output. When the hash matches, the cached translation is returned without an API call. When the hash differs, the post is re-translated and the cache entry is replaced. Both cache files live under _cache/, which is git-ignored per .gitignore, so build-time state never contaminates the repository.

We will store all derived build metadata in the sidecar manifest rather than in frontmatter. The manifest is loaded once at the start of the build, passed by reference through all parse_markdown_post calls, mutated in-place as posts are processed, and persisted once at the end via save_post_metadata. This atomic-write pattern, using a temporary file renamed into place, ensures that a failed build never corrupts the manifest. Legacy frontmatter fields from earlier builds are migrated into the manifest on first encounter and then ignored.

## Status

Accepted.

## Consequences

The authoring experience has zero bilingual friction. The author writes a single Markdown file in English, runs the build, and receives a fully translated Portuguese site. There is no second file to maintain, no translation interface to learn, and no manual synchronization to remember. Adding a new post requires editing one file in one language.

Incremental builds are efficient. The SHA-256 hashing means that running the build after editing a single post triggers exactly one Gemini API call for the changed post while serving all other translations from the local cache. A build with no content changes makes zero API calls. This matters both for build speed and for staying within Gemini's rate limits, which the translator already manages conservatively with a 90-second minimum interval between requests and a model fallback chain defined in config.py.

Content integrity is preserved unconditionally. The build system treats every file under _source/posts/ as a read-only input. Derived state lives in _cache/post-metadata.json and _cache/translation-cache.json, both git-ignored. The git history of a Markdown file reflects only editorial changes made by the author, never automated metadata updates. A catastrophic cache loss, such as deleting the _cache directory, costs only the time to re-translate all posts on the next build; no source content is affected.

The translation quality has a ceiling. Gemini produces competent Brazilian Portuguese, and the optional three-stage pipeline (translate, critique, refine) improves output, but the result is fundamentally machine-generated prose. Idiomatic nuance, cultural references, and the author's personal voice may not survive translation faithfully. There is no mechanism for per-post translation overrides; if a specific paragraph needs manual correction, the only option today is to edit the cached translation, which will be overwritten the next time the source content changes. A human translator or a hybrid workflow with post-editing would produce superior results for high-stakes content.

Content freshness is coupled to the build. Translations are generated at build time, not on demand. If the author edits a post, the Portuguese version remains stale until the build is re-run. There is no webhook, no continuous deployment trigger, and no background process watching for changes. This is acceptable for the current publishing cadence but would become a friction point if the site moved to frequent small updates.

The system depends on an external API for new content. A first build on a fresh clone, or a build after clearing the cache, requires a valid GEMINI_API_KEY and network access to the Gemini API. If the API is unavailable or the key is expired, the build fails entirely rather than producing an English-only fallback. This is a deliberate choice: a half-translated site would be worse than a failed build, since stale or missing Portuguese pages would silently degrade the experience for the primary audience.
