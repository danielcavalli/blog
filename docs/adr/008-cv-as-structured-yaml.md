# ADR 8: CV as Structured YAML

## Context

The dan.rio blog is a bilingual personal site that publishes all content in both English and Brazilian Portuguese. The CV page is one of four page types generated by the build system, alongside blog posts, the index, and the about page. Unlike blog posts, which are authored in Markdown with YAML frontmatter, the CV has a rigid, repeating structure: contact details, a summary, a list of job positions with achievements, an education section, and skills. This structure must be machine-readable so the build system can translate it field by field into Portuguese and render it into styled HTML.

The project originally maintained the CV as cv_reference.md, a hand-written Markdown file with headings, bullet lists, and inline formatting. The build system parsed this Markdown heuristically, pattern-matching headers like "## Experience" and "## Education" to extract sections. This approach was fragile. Markdown is a presentation format, not a data format: there was no schema, no validation, and any formatting change could silently break the parser. The heuristic parser also made bilingual rendering difficult, because the translator needed structured fields to produce a faithful Portuguese version rather than a free-form prose translation of the entire document.

The project's translation pipeline sends content to Gemini as structured data and expects structured data back. A flat Markdown string provides no field boundaries for the translator to respect, meaning job titles, company names, and date ranges could be reworded or reordered. The build system needs a data format where each field is individually addressable so it can translate prose fields like descriptions and achievements while preserving invariant fields like company names and dates.

## Decision

We will maintain the CV as a structured YAML file, cv_data.yaml, at the project root. This file is the single source of truth for all English CV content. It contains top-level scalar fields for name, tagline, location, and summary, a contact mapping with email, LinkedIn, and GitHub handles, and lists for skills, languages spoken, experience entries, and education entries. Each experience entry contains title, company, location, period, description, and an achievements list. Each education entry contains degree, school, and period.

We will validate cv_data.yaml at build time using a custom schema validator in cv_parser.py. The load_cv_data function reads the file with yaml.safe_load, then checks that required top-level keys exist and are non-empty, that contact contains its required sub-fields, and that every experience and education entry has its mandatory fields. Validation errors are accumulated into a list and reported together, so the author sees all problems in a single build run rather than fixing them one at a time. Missing optional fields like tagline and location produce warnings rather than errors, and optional sub-fields on experience entries receive sensible defaults. If any required field fails validation, the build exits immediately with a clear error message before any HTML generation or translation begins.

We will render the CV to HTML at build time via the generate_cv_html function in renderer.py. This function takes a language code and an optional pre-translated CV dictionary. For English, it loads cv_data.yaml directly through cv_parser.py. For Portuguese, it receives the translated dictionary produced by the Gemini translation pipeline, which translates prose fields while preserving structural invariants. The renderer iterates over experience and education entries, escaping all values and assembling HTML with the same f-string approach used throughout the project. The build orchestrator in build.py calls load_cv_data early to fail fast on validation errors, then passes the English data to the translator, and finally calls generate_cv_html twice, once per language.

## Status

Accepted.

## Consequences

Build-time schema validation catches structural errors before any content reaches production. A missing job title, an empty summary, or a malformed contact section produces a clear, accumulated error report and halts the build. This is a meaningful improvement over the previous Markdown approach, where a misplaced heading would silently produce broken HTML. The validation runs at the very beginning of the build pipeline, so errors surface instantly rather than after translation API calls.

The structured format makes bilingual rendering clean and predictable. The translator receives a dictionary of fields and returns a dictionary of the same shape, so the renderer does not need to know whether it is working with original or translated data. Invariant fields like company names and date ranges pass through untouched because the translation pipeline can distinguish them from prose fields. The cv_reference.md file is retained as an archived human-readable reference but is no longer parsed by the build.

YAML provides a git-friendly, human-editable format. Changes to the CV produce clean diffs because each field occupies its own line or block scalar, and the file's 188-line length is manageable for manual editing. The structured format also means the CV data is available for any future use beyond HTML rendering, such as generating a PDF or feeding a structured data API.

The schema validation in cv_parser.py is imperative Python code rather than a declarative schema language like JSON Schema or Pydantic models. This means the validation logic must be maintained by hand as the CV structure evolves, and it cannot be reused outside the Python build system. A declarative schema would be self-documenting and could generate validation for other tools, but it would add a dependency and a layer of abstraction to a project that deliberately minimizes both. The current approach is roughly one hundred lines of straightforward conditionals, which is proportionate to the problem.

YAML's indentation sensitivity introduces a class of editing errors that a format like JSON or TOML would not. A misaligned field in an experience entry can change the document's structure silently, though the schema validator will catch most such errors at build time. The CV is also locked into this project's build pipeline. There is no export to a cross-system format like JSON Resume, no import from LinkedIn's API, and no integration with a CMS. The author updates the YAML by hand and rebuilds. For a single-author site where the CV changes a few times a year, this manual workflow is adequate, but it would not scale to a team or a frequently updated portfolio. These are acceptable tradeoffs given the project's single-author scope and its philosophy of minimal tooling.
